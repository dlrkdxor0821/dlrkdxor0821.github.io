# 인지 모델 평가 지표 분석

> GogoPing(Pingdergarten) 카메라 인지 모델 평가를 위한 개념 노트
> 핵심 질문: "이 숫자를 믿고 로봇에 올려도 되는가?"

---

## 0. 한 장 요약

모든 detection 지표는 결국 네 가지 카운트(TP / FP / FN / TN)에서 파생된다.
이 네 개만 정확히 이해하면 precision, recall, mAP, F1 전부 자동으로 따라온다.

```
실제로 있는 것을 ─┬─ 찾았다  → TP (정답)
                 └─ 못 찾았다 → FN (놓침)  ← 안전에서 제일 위험

없는 것을 ────────┬─ 찾았다고 함 → FP (헛것 / 오탐)
                 └─ 안 찾음     → TN (정상, detection에선 거의 안 셈)
```

문장으로 외우면:

- **TP (True Positive)** — 있는 걸 맞게 찾음
- **FN (False Negative)** — 있는데 못 찾음 (**놓침**)
- **FP (False Positive)** — 없는데 있다고 함 (**헛것/오탐**)
- **TN (True Negative)** — 없는 걸 없다고 함 (detection에선 보통 무의미)

"있는데 못 찾은 거"가 FN, "있는 걸 찾은 거"가 TP — 이게 recall의 재료다.

---

## 1. Confusion Matrix (혼동 행렬)

위 네 가지를 표로 정리한 게 confusion matrix다. 모든 지표의 출발점.

|                       | 실제로 **있음** (Positive) | 실제로 **없음** (Negative) |
| --------------------- | :-----------------------: | :-----------------------: |
| 모델이 **있다**고 함  |        **TP** ✅          |        **FP** ❌ (오탐)    |
| 모델이 **없다**고 함  |        **FN** ❌ (놓침)    |        **TN**             |

- 가로로 보면 → 모델이 뭐라고 예측했나
- 세로로 보면 → 실제 정답(ground truth)이 뭐였나
- 대각선(TP, TN)이 맞춘 것, 나머지(FP, FN)가 틀린 것

---

## 2. Detection에서 TP/FP/FN을 "어떻게 판정"하나 (중요)

단순 분류(classification)와 detection의 결정적 차이가 여기 있다.
분류는 정답/오답이 깔끔하지만, detection은 **위치까지** 맞아야 하므로 판정에 두 개의 기준이 끼어든다.

### (1) IoU — 박스가 얼마나 겹치나
예측 박스와 정답 박스의 겹친 정도.

```
IoU = (교집합 넓이) / (합집합 넓이)
```

- IoU = 1.0 → 완벽히 일치
- 보통 **IoU ≥ 0.5** 이면 "위치 맞음"으로 인정 → 그 예측은 TP 후보
- 임계값보다 낮으면 → 같은 객체를 가리켜도 위치가 틀린 걸로 보고 **FP**

### (2) Confidence threshold — 얼마나 확신하나
모델이 각 박스에 붙이는 확신도(conf). 이 값이 운영 임계값(예: 0.5)보다 낮으면 그 예측은 아예 버려진다.

### 판정 흐름
1. conf < threshold → 버림
2. 남은 예측 중 정답 박스와 IoU ≥ 0.5 매칭됨 → **TP**
3. 정답에 매칭 안 된 예측 (헛것이거나 위치 틀림) → **FP**
4. 어떤 예측에도 매칭 안 된 정답 객체 (놓침) → **FN**

> 그래서 detection의 TP/FP/FN은 **IoU 임계값과 conf 임계값에 따라 값이 바뀐다.**
> "지표 하나"가 아니라 "어떤 임계값에서의 지표냐"가 항상 따라붙는 이유.

## 3. Precision & Recall

같은 재료(TP/FP/FN)로 만들지만 **묻는 질문이 다르다.**

### Precision (정밀도) — "찾았다고 한 게 진짜 맞아?"
```
Precision = TP / (TP + FP)
          = 맞게 찾은 것 / 모델이 찾았다고 한 전체
```
헛것(FP)에 벌점. **오탐을 싫어하는 지표.**

### Recall (재현율) — "있는 거 다 찾았어?"
```
Recall = TP / (TP + FN)
       = 맞게 찾은 것 / 실제로 존재하는 전체
```
놓침(FN)에 벌점. **누락을 싫어하는 지표.**

### 트레이드오프
conf threshold가 둘을 직접 흔드는 손잡이다.

- threshold ↑ → 확실한 것만 채택 → precision ↑, recall ↓
- threshold ↓ → 의심스러운 것도 채택 → recall ↑, precision ↓

극단 예시: 화면 전체를 박스로 도배하면 recall = 1.0 (다 "찾았으니까") 이지만 precision은 바닥. **그래서 recall은 절대 혼자 보면 안 되고 precision과 짝으로 본다.**

---

## 4. PR Curve → AP → mAP

### PR Curve
conf threshold를 1.0부터 0.0까지 쭉 내리면서 (precision, recall) 점을 찍어 이은 곡선.
threshold마다 두 값이 변하므로 모델의 전체 성능 그림이 된다.

### AP (Average Precision)
PR 곡선 **아래 면적**. 한 클래스에 대해 "모든 운영점을 통틀어" 얼마나 좋은지 하나의 숫자로 압축. 0~1.

### mAP (mean Average Precision)
모든 클래스의 AP를 **평균**낸 값. 우리가 흔히 보는 그 지표.

| 이름           | 의미                                                              |
| -------------- | ----------------------------------------------------------------- |
| **mAP50**      | IoU 0.5 기준 mAP. 위치가 대충 맞으면 OK. 비교적 후한 점수.          |
| **mAP50-95**   | IoU 0.5, 0.55, … 0.95 (10단계) 평균. 위치 정밀도까지 빡세게 봄.     |

> mAP50-95 < mAP50 은 **항상 정상**이다. 격차가 크다고 모델이 나쁜 게 아니라, 위치 정밀도 페널티 때문. COCO(어려운 80클래스)에선 SOTA도 mAP50-95 0.5~0.56 수준.

## 5. F1 & fitness

### F1 Score
precision과 recall의 **조화평균**. 둘 다 좋아야 높아진다 (한쪽이 낮으면 확 떨어짐).
```
F1 = 2 × (Precision × Recall) / (Precision + Recall)
```
precision/recall 균형을 한 숫자로 보고 싶을 때.

### fitness (Ultralytics)
best.pt를 고를 때 쓰는 가중합 스칼라. **독립 평가 지표가 아님.**
```
fitness = 0.1 × mAP50 + 0.9 × mAP50-95
```
90%가 mAP50-95라서 사실상 거기에 붙어다님. 따로 "몇 이상" 기준 잡을 필요 없음.

## 6. 대략적 해석 기준 (절대선 아님 / 1차 필터용)

> 칼같은 경계선은 없다. 데이터 난이도에 따라 천차만별. 아래는 "이 근처면 이런 느낌"의 감각.

| 지표        | 약함     | 실용     | 강함 / 안전용 |
| ----------- | -------- | -------- | ------------- |
| mAP50       | < 0.5    | 0.7~0.8  | 0.9+          |
| mAP50-95    | < 0.4    | 0.5+     | 0.6~0.7       |
| Recall      | < 0.7    | 0.8~0.9  | 0.95+         |
| Precision   | < 0.7    | 0.8~0.9  | 0.95+         |
| F1          | < 0.6    | 0.8+     | 0.9+          |

**용도별 우선순위 (GogoPing):**
- 안전 클래스(아이 감지, 충돌 회피) → **recall이 생명** (놓치면 사고). precision 좀 희생 OK, 목표 recall 0.95+
- 작업 트리거(장난감 식별 등) → **precision 우선** (엉뚱한 거 집으면 곤란)

## 7. ⚠️ "너무 완벽한" 지표를 의심하는 법 (pinky_pro 케이스)

`pinky_pro`: mAP50 0.995, mAP50-95 0.991, Recall 1.000, Precision 1.000

전부 만점에 가까운 건 **좋은 신호가 아니라 경고 신호**다. 현실 데이터에서 perfect score는 거의 안 나온다. 다음을 의심해야 한다.

### 의심 1 — Data Leakage (데이터 누수)
train 데이터가 val에 섞여 들어감. 모델이 "이미 본 답"을 채점받는 셈이라 만점이 나온다.
- **확인법:** train/val 이미지 파일명·해시 중복 검사. 같은 영상에서 연속 프레임을 잘라 train/val에 나눠 담았는지 점검 (거의 동일한 프레임 = 사실상 누수).

### 의심 2 — Validation 셋이 너무 쉬움 / 너무 작음
val 이미지가 몇 장 안 되거나, 배경·각도·조명이 train과 판박이면 만점이 쉽게 나온다.
- **확인법:** val 장수 확인. 실제 주행 환경과 얼마나 다른지 눈으로 확인.

### 의심 3 — Overfitting (과적합)
모델이 일반 패턴이 아니라 그 데이터셋을 통째로 "외움". val이 train과 분포가 같으면 과적합이 만점으로 위장된다.
- **확인법:** **완전히 새로 찍은** 실배포 환경 이미지로 재검증. 여기서 점수가 폭락하면 과적합 확정.

### 의심 4 — 너무 단순한 태스크
객체가 1종류뿐이거나, 배경이 단색이거나, 객체가 항상 화면 중앙에 크게 나오면 문제 자체가 쉬워서 만점이 정상일 수도 있다. (이건 실제로 쉬운 거라 문제는 아니지만, "실제 환경도 그만큼 쉬운가?"를 물어야 함)

### 의심 5 — 라벨/평가 설정 오류
val 라벨이 train과 동일 파일을 가리키거나, 평가 스크립트가 잘못 짜여 자기 자신을 채점하는 경우.

> **핵심 원칙:** val 점수는 "이 모델이 망했나"를 거르는 1차 필터일 뿐.
> **진짜 판단은 실배포 환경에서 새로 찍은 데이터로 한 2차 검증.**
> 숫자가 완벽할수록 더 의심하고, 새 데이터로 확인하라.

## 8. 전문가 체크리스트

모델 평가할 때 이 순서로 본다:

- [ ] **평균 지표 1차 필터** — mAP50 0.7+, mAP50-95 0.5+ 인가? (학습 실패 거르기)
- [ ] **지표가 비현실적으로 완벽한가?** — 그렇다면 leakage/overfitting부터 의심
- [ ] **train/val 분리 검증** — 파일 중복, 연속 프레임 누수 없나
- [ ] **클래스별 지표** — 평균에 묻힌 최약 클래스 찾기 (안전 클래스가 죽어있나?)
- [ ] **운영 conf threshold에서의 PR** — mAP(전 임계값 적분)가 아니라 실제 conf 0.5에서의 precision/recall
- [ ] **용도별 우선순위** — 안전=recall, 작업=precision
- [ ] **실배포 환경 재검증** — 새로 찍은 조명·각도 데이터로 최종 확인

## 부록 — 핵심 공식 모음

```
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1        = 2 · (P · R) / (P + R)
IoU       = 교집합 / 합집합
AP        = PR 곡선 아래 면적 (클래스별)
mAP       = 모든 클래스 AP의 평균
mAP50     = IoU 0.5 기준 mAP
mAP50-95  = IoU 0.5~0.95 (10단계) 평균 mAP
fitness   = 0.1 · mAP50 + 0.9 · mAP50-95
```
