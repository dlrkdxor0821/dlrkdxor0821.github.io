# 기술조사 - 배차 알고리즘 (task_dispatcher)
FMS 자체 제작 참고용. 교통 협상 문서의 짝 문서. 교통 협상이 "이미 배정된 로봇들이 길에서 안 부딪히게"라면, 배차는 그 앞 단계 — **"이 task를 누구한테 줄까"** 를 정함. 학계 용어로는 **MRTA (Multi-Robot Task Allocation)**.

각 절 포맷은 교통 협상 문서와 동일: ① 뭔지(구성요소) + 왜 같이 쓰나 → ② 플로우 → ③ 장단점.

## 0. 배차 지도
교통 협상이 "언제 계획 / 누가 결정" 2축이었다면, 배차는 **MRTA 택소노미(Gerkey & Matarić, 2004)** 3축으로 봄. 배차 알고리즘을 정리하는 표준 좌표.

- **ST vs MT** — 로봇 하나가 한 번에 task 1개만(Single-Task) vs 여러 개(Multi-Task)

- **SR vs MR** — task 하나에 로봇 1대면 됨(Single-Robot) vs 여러 대 필요(Multi-Robot)

- **IA vs TA** — 지금 들어온 것만 즉시 배정(Instantaneous) vs 미래 task까지 묶어 일정 최적화(Time-extended)

우리 FMS(이송/가이드/순찰 같은 단독 task)는 거의 **ST-SR**. IA냐 TA냐가 설계 선택 포인트.

| 방식 | 대표 알고리즘 | 결정 위치 | 분류 | 특징 (요약) | 상세 |
| --- | --- | --- | --- | --- | --- |
| 탐욕/효용 | Greedy + Cost Function | 중앙 | ST-SR-IA | 단순·빠름, 근시안적 — **현재 우리 방식** | §1 |
| 최적 매칭 | Hungarian (Kuhn-Munkres) | 중앙 | ST-SR-IA | 1:1 최적 보장, 일괄 | §2 |
| 경매 | Auction / SSI | 중앙/분산 | ST-SR | 확장성·내고장성, 통신 ↑ | §3 |
| 묶음 합의 | CBBA | 분산 | ST-MR-TA | 분산 최적 근사(50% 보장) | §4 |
| 수리최적화 | MILP / VRP | 중앙 | TA | 최적이지만 무거움 | §5 |
| 메타휴리스틱 | GA / ACO | 중앙 | TA | 대규모용, 보장 X | §6 |
| RMF식 | open-rmf bidding | 중앙 DB + fleet | ST-SR | 경매 기반(= 교통문서 §6) | §3 참고 |

## 1. Greedy + Cost Function (효용 기반)
task가 들어오는 대로 그 순간 가장 좋은 로봇에 즉시 배정. **현재 우리 1차 구현.**

### ① 뭔지
task가 들어오면 각 로봇에 대해 점수(비용)를 계산하고, 제일 좋은 로봇에게 즉시 배정. 우리가 쓰는 `distance + battery + task + patrol + congestion` 가중합이 바로 이 cost function. "한 번에 하나씩, 그 순간 최선"이라 IA에 해당.

**왜 cost function인가** — 단일 기준(거리만)으론 배터리 부족 로봇이 뽑히거나, 혼잡 구역 로봇이 뽑혀 교통 협상 부담이 커짐. 여러 요소를 가중합으로 묶어 한 점수로 만들면 "종합적으로 가장 적합한" 로봇을 고를 수 있음.

### ② 플로우

```
task 도착 → 가용 로봇마다 cost 계산
         → argmin(cost) 로봇에 배정 → 다음 task
```

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 구현 가장 단순 · 빠름 · 실시간 | 전역 최적 아님 · 근시안적(나중에 더 적합한 로봇 와도 늦음) |

**한 줄:** 들어오는 대로 그 순간 제일 싼 로봇한테. 우리 1차 구현으로 정답.

## 2. Hungarian (Kuhn-Munkres)
대기 task를 모아 **일괄(batch)로 최적 1:1 매칭.** Greedy의 바로 다음 업그레이드.

### ① 뭔지
N개 로봇 × M개 task의 **비용 행렬**을 만들고, 총비용이 최소가 되는 **1:1 완전 매칭**을 한 번에 푸는 알고리즘. Greedy가 task 하나씩 탐욕적으로 보는 것과 달리, 여러 task를 모아 동시에 최적 배정. 핵심은 "일괄(batch)로 최적".

**왜 Greedy 대신** — Greedy는 task A를 로봇1에 줬는데, 직후 들어온 task B에는 로봇1이 훨씬 더 적합한 경우를 되돌리지 못함(근시안). Hungarian은 A·B를 모아 행렬로 보고 전체 합이 최소가 되게 배정하므로 이런 손해를 피함.

참고 — task 수 ≠ 로봇 수면 가상(dummy) 행/열로 패딩해 정사각 행렬을 만든 뒤 풂.

### ② 플로우

```
대기 task 모음 → cost 행렬 [robot × task] 구성
             → Hungarian으로 최소비용 매칭 → 일괄 배정
```

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 그 배치(batch)에 대해 최적 보장 · O(n³) | task를 모아야 효과 ↑ · 순서/시간(TA) 미반영 · 패딩 필요 |

**한 줄:** cost 행렬 짜서 최적 1:1 매칭. Greedy의 바로 다음 업그레이드.

## 3. Auction / SSI (경매 기반)
task를 경매에 부쳐 최저가 입찰자가 낙찰. **open-rmf가 바로 이 방식**(교통문서 §6).

### ① 뭔지
**Auction(경매)** — task를 경매에 부치면 각 로봇이 "내가 하면 비용 이만큼"이라고 **입찰**하고, 최저가가 낙찰. 중앙 없이도 굴러가 분산화에 강함.

**SSI (Sequential Single-Item)** — task를 한 개씩 라운드로 돌려 매번 입찰→낙찰을 반복하는 변형. 한 라운드마다 가장 이득 큰 1개만 배정하고 다음 라운드로 넘어감. 단순함과 품질의 균형이 좋아 실무에서 많이 씀.

**왜 같이 쓰나** — 순수 경매는 한 번에 모든 task를 부치면 입찰 계산이 복잡해짐. SSI는 "한 라운드 한 개"로 쪼개 계산을 단순화하면서도, 라운드마다 최신 상태를 반영해 품질을 유지함.

### ② 플로우

```
task 공고(announce) → 로봇들 bid(cost) 제출
                   → 최저가 낙찰 → (SSI면 다음 task 라운드 반복)
```

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 분산 가능 · 로봇 추가/고장에 강함 · 확장성 | 라운드 간 탐욕이라 전역 최적 X · 통신 오버헤드 |

**한 줄:** "이 일 할 사람?" → 최저가 낙찰. open-rmf가 바로 이 방식.

## 4. CBBA (Consensus-Based Bundle Algorithm)
분산 + 다중 task용 정석. 각자 일감 바구니 담고, 겹치면 합의로 양보.

### ① 뭔지
분산 환경에서 **여러 task를 묶음(bundle)** 으로 배정하는 알고리즘. 2단계로 돎.

- **bundle construction(묶음 구성)** — 각 로봇이 자기에게 이득 큰 task를 탐욕적으로 자기 bundle에 담음.

- **consensus(합의)** — 로봇끼리 입찰값을 주고받아 "이 task는 누구 거"를 합의로 충돌 해소. 더 높은 입찰자가 차지하고, 진 로봇은 그 task를 bundle에서 뺌.

중앙 지휘 없이 분산으로 최적 근사(이론상 최적의 50% 이상 보장)에 도달하는 게 특징. ST-MR-TA까지 커버.

**왜 같이 쓰나** — bundle만 있으면 여러 로봇이 같은 task를 중복으로 담아 충돌함. consensus 단계가 이 충돌을 분산 합의로 정리해줘야 비로소 모순 없는 배정이 됨.

### ② 플로우

```
[bundle 단계] 각 로봇 탐욕적으로 task 묶음 구성
[consensus 단계] 이웃과 입찰값 교환 → 충돌 task 양보 → 수렴까지 반복
```

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 완전 분산 · 수렴/품질 보장 · 다중 task | 구현 복잡 · 통신 라운드 필요 |

**한 줄:** 각자 일감 바구니 담고, 겹치면 합의로 양보. 분산 + 다중 task용 정석.

## 5. 수리최적화 (MILP / VRP)
문제를 수식으로 만들어 솔버에 던지기. 규모 작고 최적이 중요할 때.

### ① 뭔지
**MILP (혼합정수계획)** — 배차를 정수계획으로 모델링해서 솔버로 최적해를 구함. "어느 로봇이 어느 task를 맡을지"를 0/1 변수로, "총 이동·지연 최소"를 목적함수로, "용량·시간창"을 제약으로 표현.

**VRP / mTSP** — task가 pickup-delivery + 경로(이송 동선)까지 엮이면 **VRP(Vehicle Routing Problem)** / multiple TSP 형태로 푸는 게 자연스러움. 시간·순서까지 고려하는 TA의 대표.

**왜 같이 쓰나** — 단순 배정만이면 MILP 매칭으로 충분하지만, "한 로봇이 여러 지점을 어떤 순서로 도느냐"까지 최적화하려면 VRP 모델이 필요. 둘은 같은 수리최적화 틀의 난이도 차이.

### ② 플로우

```
목적함수(총 이동/지연 최소) + 제약(용량·시간창) 정식화
                                → 솔버(예: OR-Tools)로 최적/근사 해
```

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 전역 최적 · 복잡 제약 표현 가능 | 무거움(NP-hard) · 실시간 어려움 · 규모 ↑ 폭발 |

**한 줄:** 문제를 수식으로 만들어 솔버에 던지기. 규모 작고 최적이 중요할 때.

## 6. 메타휴리스틱 (GA / ACO / Tabu)
최적 포기, 규모로 승부. (FMS 초기 규모엔 보통 오버스펙)

### ① 뭔지
MILP가 너무 무거운 대규모에서 쓰는 근사법. 유전 알고리즘(GA), 개미군집(ACO), 타부 서치(Tabu Search) 등으로 "충분히 좋은" 해를 빠르게 찾음. 최적 보장은 없지만 규모 확장성이 좋음.

- **GA** — 배정안을 "유전자"로 보고 교배·돌연변이로 좋은 해를 진화시킴.

- **ACO** — 개미가 페로몬을 남기듯 좋은 경로/배정에 가중치를 쌓아 수렴.

### ③ 장단점

| 장점 | 단점 |
| --- | --- |
| 대규모에 현실적 · 유연 | 최적 보장 X · 파라미터 튜닝 의존 |

**한 줄:** 최적 포기, 규모로 승부. FMS 초기 규모엔 보통 오버스펙.

## 전체 비교표

|  | Greedy + Cost | Hungarian | Auction / SSI | CBBA | MILP / VRP |
| --- | --- | --- | --- | --- | --- |
| **결정 시점** | task마다 즉시 | batch 모아서 | task/라운드마다 | 라운드 합의 | 일괄 정식화 |
| **결정 위치** | 중앙 | 중앙 | 중앙/분산 | 분산 | 중앙 |
| **MRTA 분류** | ST-SR-IA | ST-SR-IA | ST-SR | ST-MR-TA | TA |
| **최적성** | 근시안(국소) | batch 내 최적 | 라운드 탐욕 | 50% 근사 보장 | 전역 최적 |
| **확장성** | 높음 | 중간(n³) | 높음(분산) | 높음(분산) | 낮음(폭발) |
| **구현 난이도** | 낮음 | 중간 | 중간 | 높음 | 높음 |
| **실시간성** | 강함 | 중간 | 강함 | 중간 | 약함 |

### 배차 ↔ 교통 협상 관계

```
Task Manager → Scheduler → Dispatcher → Fleet Adapter → Robot
                  └─────────┬────────┘        └────────┬───────┘
                       이 문서(배차)          교통 협상 문서(주행 중)
```

Scheduler/Dispatcher 자리가 **이 배차 알고리즘**, Fleet Adapter 이후 주행 단계가 **교통 협상**. 우리 OpenRMF식 파이프라인(Task Manager → Scheduler → Dispatcher → Fleet Adapter → Robot)에 그대로 대응됨.

### 우선순위 연결
교통 협상 문서 §5에서 정리한 task 우선순위(이송 > 가이드 > returning > 파견)는 배차에도 그대로 입력 가능 — cost function의 `task` 가중치 항이 바로 그 우선순위를 점수화하는 자리. 즉 **배차의 cost 항 ↔ 교통 협상의 양보 순서**가 같은 우선순위 정책을 공유함.
