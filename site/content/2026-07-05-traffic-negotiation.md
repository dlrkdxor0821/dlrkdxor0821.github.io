# 기술조사 - 교통 충돌 시 협상 알고리즘
FMS 자체 제작 참고용. 각 절은 동일 포맷: **① 뭔지(구성요소) + 왜 같이 쓰나 → ② 플로우 → ③ 장단점**.

## 0. 충돌 회피 알고리즘 지도

| 방식 | 대표 알고리즘 | 결정 위치 | 특징 (요약) | 상세 |
| --- | --- | --- | --- | --- |
| **중앙 일괄계획** | CBS + space-time A* | 중앙 | 최적 보장, 무거움 — *예: robocallee* | §1 |
| **예약 + 데드락 해소** | Dijkstra + 노드예약 + DFS | 중앙/혼합 | 실행 중 예약·순환 감지 양보 | §2 |
| **구간 잠금** | Edge/Node Lock, lane lock | 중앙(서버) | 단순, 교착 관리 쉬움 — *예: 쑈뼁끼* | §3 |
| **상호 양보 (속도)** | ORCA / VO (RVO) | 분산(로봇) | 분산·실시간·빠름, 교착 위험 | §4 |
| **우선순위 협상** | VR-ORCA, MCCA, priority-based | 분산(로봇) | 양보 순서 결정, 교착 완화 | §5 |
| **RMF식 협상** | open-rmf negotiation | 중앙 DB + fleet | schedule DB 필요, 정교 | §6 |

갈림 축 2개: **언제 계획**(출발 전 ↔ 실행 중) · **누가 결정**(중앙 ↔ 분산). "교착 관리 쉬움" = 구조가 단순해 *순서화로 예방*하기 쉽다는 뜻(자동 해소는 아님, §3).

## 1. CBS + space-time A*
중앙에서 **출발 전 모든 경로를 미리** 계산 → 충돌 0으로 동시 출발.

**① 뭔지**

- **CBS (Conflict-Based Search)** — 모든 로봇 경로를 한 번에 똑똑하게 짜는 게 아니라, *일단 각자 남 신경 안 쓰고 최단경로*를 그린 뒤, **부딪히는 지점이 보이면 "그 로봇은 그 시각·그 위치 가지 마"라는 규칙(제약)을 하나 추가**하고 그 로봇만 다시 그리게 함. 충돌이 없어질 때까지 "규칙 추가 → 재계산"을 반복하는 **심판** 역할. (경로를 직접 만들진 않음 — 그건 A* 몫)

- **space-time A*** — 보통 A*는 "어느 칸으로 갈까"(공간)만 푸는데, 여기에 **시간 축을 더해서 "몇 초에 어디 있을까"까지** 같이 푸는 길찾기. 위치만이 아니라 `(위치, 시각)`을 다루니까 **"여기서 한 박자 쉬었다 가기(wait)"** 같은 것도 자연스럽게 경로의 일부가 됨. CBS가 준 금지 규칙을 지키면서 우회로를 찾는 **선수** 역할.

참고 — 부딪힘(충돌)엔 2종류: **vertex**(같은 시각·같은 위치에 둘이 있음) / **edge**(서로 자리를 맞바꾸며 지나침).

**왜 같이 쓰나** — CBS는 *충돌을 찾아 규칙만* 주고, A*는 *그 규칙을 지키며 실제 길을* 그림. **"누가 부딪혀? (CBS)" + "그럼 어떻게 비켜? (A*)"** 가 한 팀. 하나만으론 안 됨 — CBS 혼자선 길을 못 만들고, A* 혼자선 다른 로봇 존재를 모름.

**② 플로우**

```
1. 각 로봇 독립 최단경로
2. 충돌 검사 (vertex / edge)
3. 충돌 → 한쪽에 "(위치, t) 금지" 제약 → 그 로봇만 A* 재계산
4. 충돌 없을 때까지 2~3 반복
5. 완성 경로 배포 → 동시 출발

```

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 출발 시 충돌 0 보장(최적) · 실시간 통신 불필요 | 동적 장애물 막히면 **전체 재계획** · 로봇↑ 계산량 폭발 |

한 줄: 출발 전 체스처럼 다 계산, 막히면 처음부터.

## 2. Dijkstra + 노드 예약 + 데드락 DFS
**실행 중 실시간**으로 충돌 회피 (사전 계획 X).

**① 뭔지**

- **Dijkstra** — 가장 기본적인 **최단경로 찾기** 알고리즘. 여기선 길이 막히면 *돌아가는 길*도 다시 찾아주는 **내비게이션** 역할.

- **노드 예약** — 로봇이 지나갈 길목(노드)을 **미리 "나 몇 시~몇 시에 여기 쓸게" 하고 잠가두는 것**. 그래야 다른 로봇이 같은 자리에 동시에 못 들어옴. **주차 예약**과 같은 개념.

- **데드락(교착) DFS** — "A는 B가 비키길, B는 A가 비키길" 처럼 **서로 기다리다 둘 다 멈춰버린 상황(교착)**을 찾아내는 장치. *누가-누굴-기다리는지* 화살표 그래프를 그렸을 때 **빙 도는 고리(사이클)가 생기면 = 교착**이라고 판정함(그 고리를 찾는 데 DFS 탐색을 씀) → 고리 안의 낮은 우선순위 로봇을 *후퇴/주차*시켜 고리를 끊음. **교통경찰** 역할.

**왜 같이 쓰나** — Dijkstra만 있으면 남이 어디 있는지 몰라 부딪히고, 예약만 있으면 서로 양보를 안 해 **교착**이 남음. 그래서 **길찾기(Dijkstra) + 자리잠금(예약) + 교착풀기(DFS)** 셋이 한 세트.

⚠️ 양보시킨 뒤에도 같은 교착이 반복(livelock)될 수 있어서, 우선순위는 매번 같은 기준으로 **고정**해 둬야 함.

**② 플로우**

```
이동 시작 → Dijkstra 경로 → 노드 예약
   겹치면 → 대기 or 우회
A 통과 → 예약 해제 → 다음 로봇 진입
순환 대기 감지(DFS) → 낮은 우선순위 양보

```

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 동적 장애물에 비교적 강함 (그때그때 대응) | 충돌 0 보장 X · 데드락 해소 후 재시도 |

한 줄: 달리면서 자리 잠그고, 막힌 순환은 DFS로 풀기.

## 3. Edge/Node Lock (Request–Approve–Release)
**한 칸 갈 때마다 서버에 허락**받고 이동. 본질은 **분산 락(mutex)**.

**① 뭔지**

- **Edge Lock (구간 잠금)** — 한 지점에서 다음 지점으로 가는 **길(통로) 자체를 잠그는 것**. "이 복도 지금 A가 지나가는 중이니 들어오지 마." → 통로 한가운데서 마주치는 충돌을 막음.

- **Node Lock (도착지 잠금)** — **도착할 자리(목적 WP)를 잠그는 것**. "이 칸은 A가 들어갈 거니까 비워둬." → 같은 칸에 둘이 들어가려는 충돌을 막음.

- **FIFO 큐** — 같은 자리를 기다리는 로봇이 여럿이면 **온 순서대로 줄 세우는 번호표**. 없으면 누구를 먼저 보낼지 모르고, 특정 로봇만 계속 밀릴 수 있음.

**왜 같이 쓰나** — 통로(edge)만 잠그면 도착지 충돌을 못 막고, 도착지(node)만 잠그면 가는 길 충돌을 못 막음 → **둘 다** 잠가야 함. 거기에 큐로 순서까지 보장. 본질은 컴퓨터 OS의 **락(mutex, 한 번에 하나만 쓰게 하기)** 과 똑같음.

**② 플로우**

```
A: "WP3 → WP4 가도 돼요?"
서버: edge·node 비었나?
   비었음 → APPROVE → 이동
   잠김   → WAIT (FIFO 큐)
통과 → RELEASE → 큐 다음 로봇 APPROVE

```

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 구현 가장 단순 · 서버 부하 낮음 | 데드락 감지 없음 — 순환 대기 시 영원히 멈춤 |

데드락 회피책: **자원 순서화**(정해진 순서로만 락 → 순환 불가) · 교차로 단위 락 · 타임아웃. 다 붙이면 §2로 진화. 한 줄: "가도 돼요?" 묻고 가기. 단순하지만 교착은 스스로 못 풂.

## 4. ORCA / VO (상호 양보 — 속도공간 분산)
중앙 없이 **각 로봇이 매 순간 "안 부딪힐 속도"를 스스로 선택**. (§1~3은 그래프 노드 단위, 이건 **연속 속도 벡터** 단위 — 결이 다름)

**① 뭔지** — VO → RVO → ORCA, 한 계열의 발전사

- **VO (Velocity Obstacle, 속도 장애물)** — 상대 로봇의 위치·속도를 보고 **"내가 이 속도로 가면 곧 부딪힌다"는 위험한 속도들의 범위**를 계산함. 그 범위만 피해서 속도를 고르면 안전. *경로가 아니라 매 순간 "속도"를 고른다*는 게 핵심.

- **RVO (Reciprocal VO, 상호 VO)** — VO엔 문제가 있음: 둘 다 똑같이 피하면 다음 순간 또 마주봐서 좌우로 **왔다갔다 진동**함. 그래서 *"상대도 절반은 피해줄 것"* 이라 믿고 **나는 절반만** 틀게 함 → 진동이 사라짐.

- **ORCA (Optimal Reciprocal Collision Avoidance)** — RVO를 **계산하기 쉬운 형태로** 다듬은 것. "피해야 할 속도 영역"을 **직선으로 자른 반쪽 영역(반평면)** 으로 표현해서, 주변에 로봇이 많아도 **간단한 수식(선형계획, LP)으로 한 번에** 가장 적절한 속도를 구함. 실제 라이브러리 **RVO2** 로 널리 쓰임.

**왜 이렇게 발전?** — 셋은 별개가 아니라 **같은 아이디어의 1→2→3세대**: VO(진동이 단점) → RVO("상호 절반 양보"로 진동 해결) → ORCA(그걸 빠른 수학으로 실용화).

**② 플로우** (매 제어주기 반복)

```
주변 로봇 위치·속도 관측
→ 상대마다 "충돌 속도영역" = 반평면 제약
→ 모든 제약 만족 + 원하는 속도에 가장 가까운 v 를 LP로 선택
→ 그 속도로 한 스텝 → 반복

```

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 완전 분산(통신 거의 X) · 부드럽고 즉각 반응 → 동적 장애물 강함 | 충돌 0 보장 X(국소) · **좁은 통로 대칭서 데드락** · 위치/속도 추정에 민감 |

대칭 데드락(둘 다 똑같이 비키다 멈춤)이 약점 → 해소는 **§5 우선순위 협상**. 한 줄: 각자 매 순간 안 부딪힐 속도를 고름. 빠르지만 좁은 데서 서로 막힘.

## 5. 우선순위 협상 (priority-based / VR-ORCA / MCCA)
§4 ORCA의 **대칭 데드락**(둘 다 똑같이 양보 → 막힘)을 **우선순위·책임 비대칭**으로 푸는 계열. 핵심 = **"대칭을 깬다."**

**① 뭔지** — "누가 먼저 갈지" 정하는 3가지 방식

- **priority-based (우선순위 기반)** — 가장 단순·직관적. **로봇(또는 작업)마다 우선순위를 매겨두고, 높은 쪽이 그냥 직진하고 낮은 쪽이 비킴.** 응급차가 먼저 가고 나머지 차가 갓길로 빠지는 것과 같음.

- **VR-ORCA** — ORCA에서 "둘 다 절반씩 양보"였던 **회피 책임을 상황에 맞게 다르게** 나눔. 50:50 고정이 아니라 "둘이 합쳐서 100%면 됨"으로 풀어서, 더 급한 쪽은 조금만 비키고 덜 급한 쪽이 더 많이 비키게 함.

- **MCCA** — **'masked velocity'(가려진 속도)** 라는 *숨은 우선순위 신호*를 각 로봇이 들고, 그걸 **옆 로봇에게 퍼뜨려** "누가 먼저"인지 줄을 세움. 중앙 지휘 없이 분산으로 순서가 정해지는 게 특징(차륜형 로봇·좁은 통로에 강함).

**왜 필요** — ORCA(§4)는 "둘 다 똑같이 양보"라서 좁은 데서 서로 비키다 **둘 다 멈춤**. **누가 먼저 갈지**만 정해주면 그 교착이 풀림 → 그래서 "비대칭(우선순위)"을 일부러 넣는 것.

**⭐ 우리 FMS 연결** — task 우선순위(**이송 > 가이드 > returning > 파견**)가 그대로 입력. 이송 = 직진, 파견/returning = 양보. 분산이면 그 우선순위를 *VR-ORCA 책임비율* 또는 *MCCA masked velocity* 로 인코딩.

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 대칭 데드락 직접 해소 · "중요한 일 먼저" 정책 반영 | 우선순위 설계 나쁘면 낮은 쪽 **기아(starvation)** |

출처: [VR-ORCA (IEEE RA-L 2021)](https://ieeexplore.ieee.org/document/9384148/) · [MCCA (IEEE 2024)](https://ieeexplore.ieee.org/document/10414179/) / [arXiv:2305.04511](https://arxiv.org/abs/2305.04511) 한 줄: 우선순위로 한쪽 먼저 보내 교착 풀기 — 우리 task 우선순위와 직결.

## 6. RMF식 협상 (open-rmf)
오픈소스 **open-rmf** 의 공식 방식. 배차 + 교통을 시스템 레벨로.

**① 뭔지**

- **Bidding (배차)** — 일감이 들어오면 중앙(Dispatcher)이 **"이 일 할 사람?" 하고 공고**하고, 각 로봇 그룹(fleet)이 **"제가 하면 비용 이만큼이요" 하고 견적**을 냄 → **가장 싸게 부른 쪽이 낙찰**. 경매와 같은 방식.

- **Traffic Schedule DB** — 모든 로봇이 **"나는 앞으로 이렇게 움직일 예정" 이라는 궤적을 미리 등록**해두는 공용 일정표. 칸(격자) 단위가 아니라 **연속시간 곡선(spline)** 이라 "몇 초 뒤 정확히 어디 있을지"까지 알 수 있어 정밀함.

- **Negotiation (교통 협상)** — 일정표(DB)가 **두 로봇 궤적이 겹칠 것 같으면 미리 감지**해 양쪽에 알림 → 양쪽이 *대안 궤적*을 내고, 안 맞으면 또 내고를 반복 → 심판(**Judge**)이 **전체 지연이 가장 적은 안**을 골라줌.

- **traffic_blockade** — 협상할 능력이 없는 단순한 로봇을 위한 것. 협상 결과만 받아 **"멈춰 / 가" 신호만** 따름.

**왜 같이 쓰나** — *미리 궤적 공유(예방) → 겹침 감지 → 양방향 협상* 의 3단계로 굴러감. 큰 특징은 **우선순위가 없다**는 것 — 누가 더 중요하냐가 아니라 **전체 지연이 최소가 되도록** 판정함.

**② 플로우**

```
[배차] task → BidNotice → BidProposal(비용) → 최저가 DispatchRequest
[교통] 궤적 DB 등록 → 겹침 감지 → 협상(대안 반복 제시) → Judge 판정

```

**③ 장단점**

| 장점 | 단점 |
| --- | --- |
| 연속시간이라 정교 · 검증된 오픈소스 · fleet 간 공정 | schedule DB 등 인프라 무거움 · **우선순위 없음** → 우리 task 우선순위와 상충 |

우리와 차이: open-rmf는 동등 대우(지연 최소), 우리는 task 타입 우선순위 → **설계 선택 포인트**.

## 전체 비교표

|  | CBS + A* | Dijkstra + 예약 + DFS | Edge/Node Lock | ORCA / VO |
| --- | --- | --- | --- | --- |
| **계획 시점** | 출발 전 전체 | 실행 중 경로 단위 | 실행 중 한 칸씩 | 실행 중 매 순간(속도) |
| **결정 위치** | 중앙 | 중앙/혼합 | 중앙(서버) | 분산(로봇) |
| **단위** | 노드+시간 | 노드+시간창 | 노드/엣지 | 연속 속도 벡터 |
| **동적 장애물** | 취약(전체 재계획) | 비교적 강함 | 중간(WAIT 재시도) | 강함(즉각 반응) |
| **구현 난이도** | 높음 | 중간 | 낮음 | 중간 |
| **데드락** | 없음(사전 보장) | DFS로 해소 | 감지 안함 | 대칭 시 발생(→§5) |
| **충돌 보장** | 출발 시 보장(최적) | 보장 못함 | 보장 못함 | 보장 못함(국소) |
| **서버 부하** | 높음 | 중간 | 낮음 | 낮음(분산) |

§5 우선순위 협상 = ORCA에 우선순위를 더한 확장, §6 RMF = 시스템 레벨 방식이라 위 표(저수준 4종)와 별도.
