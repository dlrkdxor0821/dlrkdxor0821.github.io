# OCR 학습 노트

광학 문자 인식(OCR) 개념을 개발자 관점에서 정리. OCR이 뭔지, 실제 파이프라인 4단계, 인식 모델 내부 구조, 단순 텍스트를 넘어선 Document AI, 정확도 지표(CER/WER), 전처리의 역할과 "딥러닝 OCR은 과한 전처리가 독"이라는 반전, 클라우드 vs 온디바이스 선택까지.

---

## OCR이 뭐냐

**OCR(Optical Character Recognition, 광학 문자 인식)** = 이미지 속 글자를 컴퓨터가 다룰 수 있는 **텍스트 데이터**로 바꾸는 기술.

예: 영수증 사진의 "총액 12,000원"이라는 **글자 픽셀** → `"12000"`이라는 **문자열**.

쓰임: 문서 스캔, 영수증·명함 자동 입력, 번호판 인식, 사진 속 외국어 번역, 공장 라벨·일련번호 읽기 등.

## 실제 파이프라인은 4단계

개념적으로는 "검출 + 인식" 2단계지만, 실무에선 전처리·후처리가 정확도를 좌우한다.

```
① 전처리  →  ② 검출(Detection)  →  ③ 인식(Recognition)  →  ④ 후처리
```

| 단계 | 하는 일 |
|---|---|
| ① 전처리 | 흑백화, 이진화, 기울기 보정, 노이즈 제거, 해상도 키우기 |
| ② 검출 | 글자가 **어디 있는지** 박스로 찾기 (DBNet, EAST 등) |
| ③ 인식 | 박스 안 글자가 **무슨 글자인지** 읽기 |
| ④ 후처리 | 정규식·사전으로 교정, 신뢰도 필터링, 포맷 검증 |

→ ②가 "글자 위치 찾기"(탐지), ③이 "그 글자가 ㄱ인지 A인지"(분류). 객체탐지와 비슷한 흐름.

## 인식 모델 내부 구조

| 방식 | 핵심 | 특징 |
|---|---|---|
| **CRNN + CTC** | CNN으로 특징 → RNN으로 순서 → CTC로 정렬 | 고전·표준, 가볍고 빠름 |
| **Attention/Transformer** | 문맥 보고 글자 예측 | 정확도 높음, 무거움 |
| **TrOCR** | 이미지→텍스트를 통째로 | 손글씨·어려운 글자에 강함 |

> **CTC**(Connectionist Temporal Classification): 글자 개수·위치를 미리 안 정해도 **가변 길이 텍스트**를 읽게 해주는 핵심 기법. OCR 공부하면 자주 나오는 용어.

## OCR은 텍스트만 주는 게 아니다 — Document AI

실무에서 진짜 중요한 건 **글자 + 위치 + 구조**다.

- **Bounding box** — 각 글자/단어 좌표
- **Confidence score** — 인식이 얼마나 확실한지 (필터링 필수)
- **Layout analysis** — 제목/문단/표 구분
- **Table extraction** — 표를 행·열 구조로
- **Key-Value extraction** — "이름: 홍길동"에서 키-값 짝짓기 (서류 자동화의 핵심)

→ 단순 글자 읽기를 넘어서면 이걸 **Document AI / IDP(지능형 문서 처리)** 라고 부른다.

## 정확도 지표 (필수)

| 지표 | 의미 |
|---|---|
| **CER** (Character Error Rate) | **글자** 단위 오류율 — OCR 표준 |
| **WER** (Word Error Rate) | **단어** 단위 오류율 |

> "정확도 95%"는 모호하다. **CER/WER로 측정**해야 비교가 된다 (낮을수록 좋음).

## 전처리 — 무엇을, 왜

OCR 모델은 깨끗하고 또렷한 글자에 강하다. 전처리는 모델이 읽기 쉽게 이미지를 다듬는 작업.

| 단계 | 뭐 하나 | 왜 |
|---|---|---|
| Grayscale | 컬러 → 흑백 | 색은 불필요, 연산↓ |
| Resize/Upscale | 작은 글자 키우기 | 작으면 못 읽음 |
| Denoise | 잡티 제거 | 점을 글자로 오인 방지 |
| Contrast(CLAHE) | 명암 키우기 | 흐릿한 글자 또렷하게 |
| **Binarization(이진화)** | 흑/백 2색으로 | 글자와 배경 분리 — 전처리 핵심 |
| **Deskew** | 기울기 보정 | 기울면 인식률 급락 |
| Perspective 보정 | 비스듬한 걸 정면으로 | 영수증·간판 |

이진화는 조명 균일하면 **Otsu**, 그림자·불균일하면 **Adaptive threshold**.

## ⚠️ 반전: 요즘은 전처리를 덜 한다

| | 옛날 OCR (Tesseract) | 요즘 딥러닝 OCR (PaddleOCR 등) |
|---|---|---|
| 전처리 | 많이 해야 잘 됨 (이진화 필수) | 모델이 잡티·기울기를 어느 정도 알아서 처리 |
| 입력 | 흑백·이진화 선호 | **컬러 원본 그대로**가 더 나을 때 많음 |

→ 딥러닝 OCR에 과하게 이진화하면 오히려 정보가 날아가 정확도가 떨어질 수 있다.

**실무 원칙:**
1. 일단 **원본 그대로 돌려본다**
2. 안 되면 **뭐가 문제인지 진단** (어두움? 기울었나? 작나?)
3. **그 문제에 맞는 전처리만** 적용 (다 하지 말고)
4. **CER로 전/후 비교** — 추측 말고 숫자로

> premature optimization은 만악의 근원. 전처리도 **문제를 본 다음에** 최적화.

## 클라우드 API vs 온디바이스

| 기준 | 클라우드 API | 자체/온디바이스 |
|---|---|---|
| 정확도 | 보통 더 높음 | 모델·튜닝에 달림 |
| 비용 | 호출당 과금 | 초기 구축 후 무료 |
| 개인정보 | 외부 전송 ⚠️ | 로컬 처리(안전) |
| 오프라인 | 불가 | 가능 |

→ 민감 정보·오프라인 → 자체 / 빠른 개발·최고 정확도 → 클라우드(CLOVA, Google Vision).

온디바이스 배포 시 PaddleOCR 등을 **NCNN/TFLite로 변환**. 단 OCR은 검출+인식 두 모델이라 무거워서, 엣지에선 **경량 모델 + 양자화**가 거의 필수.

## 적절한 모델 선정

| 도구 | 언제 쓰면 좋나 |
|---|---|
| **EasyOCR** | 빠르게 프로토타입 — 파이썬 몇 줄로 바로 테스트하고 싶을 때 |
| **PaddleOCR** | 한글 포함 다국어, 실전 정확도와 엣지 배포(NCNN/TFLite 변환)까지 노릴 때 |
| **Tesseract** | 인쇄체 영문 문서 중심, 자료·레퍼런스 많은 검증된 표준이 필요할 때 |
| **TrOCR** | 손글씨나 까다로운 글자를 Transformer로 정확히 읽어야 할 때 |
| **CLOVA OCR / Google Vision** | 개발 부담 없이 최고 정확도 — 서류·영수증 자동화(클라우드 과금 감수) |

## 배포 런타임 (온디바이스)

OCR도 모델이라, 기기에 올리려면 **배포 런타임**으로 변환해서 얹어야 한다 (NCNN·TFLite 등이 여기 속함). OCR은 **검출 + 인식 = 모델 2개**라, 둘 다 변환·탑재해야 한다는 점만 주의.

| 런타임 | 만든 곳 | 가장 강한 타겟 | OCR 관점 메모 |
|---|---|---|---|
| **Paddle Lite** | Baidu | 모바일·ARM 엣지 | **PaddleOCR 순정 경로**, 변환 손실 적고 궁합 최고 |
| **Paddle Inference** | Baidu | 서버·데스크톱 GPU | PaddleOCR을 서버에서 돌릴 때 |
| **NCNN** | Tencent | ARM CPU·임베디드 | 가볍고 의존성 0, ONNX 거쳐 변환 |
| **TFLite** (LiteRT) | Google | 안드로이드·NPU/Edge TPU | 안드로이드 앱 통합 매끄러움 |
| **ONNX Runtime** | Microsoft | 서버·데스크톱·범용 | 플랫폼 호환성 넓음, 기준점 |
| **OpenVINO** | Intel | 인텔 CPU·iGPU·VPU | 인텔 하드웨어면 가속 큼 |
| **TensorRT** | NVIDIA | NVIDIA GPU·Jetson | GPU 극한 성능, Jetson 로봇이면 강력 |
| **MNN** | Alibaba | 멀티플랫폼 균형 | CPU·GPU 골고루, iOS·안드로이드 |
| **Core ML** | Apple | iOS·맥 | 애플 기기 전용이면 |

### PaddleOCR 변환 경로

```
PaddleOCR 모델 (검출 + 인식)
   ├─ Paddle Lite 로 변환  → 모바일·ARM (순정·1순위)
   └─ ONNX 로 export
        ├─ NCNN 으로 변환     → ARM 경량
        ├─ TFLite 로 변환     → 안드로이드
        ├─ ONNX Runtime 실행  → 서버·범용
        ├─ OpenVINO 로 변환   → 인텔
        └─ TensorRT 로 변환   → NVIDIA GPU/Jetson
```

### 타겟 기기별 추천

| 배포 대상 | 1순위 런타임 |
|---|---|
| 라즈베리파이 (ARM CPU) | **Paddle Lite** (또는 NCNN) |
| NVIDIA Jetson (로봇·GPU) | **TensorRT** |
| 안드로이드 폰 | **TFLite** 또는 Paddle Lite |
| iOS 폰 | **Core ML** 또는 MNN |
| 인텔 미니PC | **OpenVINO** |
| 서버·클라우드 | **ONNX Runtime** / Paddle Inference |

> OCR도 "학습 → 변환 → 런타임에 얹어 배포"라는 같은 흐름. **PaddleOCR이면 순정 Paddle Lite가 1순위**, 그 외엔 ONNX로 export 후 타겟 하드웨어에 맞는 런타임으로 변환.

## 흔한 함정

- 전처리 무시하고 모델만 탓하기 → 전처리부터 점검
- confidence score 안 거르기 → 쓰레기 결과까지 그대로 사용
- 비슷한 글자(O/0, l/1) → 후처리 정규식·사전으로 교정
- 한글에 Tesseract 기본값 → PaddleOCR 권장

## 핵심 요약

> OCR = 이미지 속 글자 → 텍스트. 실무는 전처리·검출·인식·후처리 4단계.
> 글자뿐 아니라 **위치·구조·신뢰도**까지 다루면 Document AI.
> 정확도는 **CER/WER**로 측정. 딥러닝 OCR은 **일단 원본으로 해보고** 안 될 때 문제에 맞는 전처리만 추가.
