# Odometry 정리 — 시작점 기준 상대 위치 추정

> **Odometry는 시작점을 기준으로 상대적인 위치를 측정하는 것.**
> (출처: SMG0317-Odometry 교육자료 + ROS2 개발자 보강)
> 흐름: **정의 → 모션 모델(rot–trans–rot) → 노이즈/드리프트 → ROS odom frame & TF → 보강 개념**

---

## Part 0. 한 줄 정의

모터(바퀴)에 달린 **엔코더의 회전수**와 **IMU(관성 측정 장비)의 기울기** 등을 측정하여, **시작점(0,0)을 기준으로 지금 얼마나 움직였는지** 상대 위치를 추정하는 방법.

> = **추측 항법(Dead Reckoning)**: *"(0,0)에서 초당 1m로 3초 직진했으니 지금 (3,0)일 거야"* 라고 **계산으로** 위치를 추정.

---

## Part 1. 오도메트리 모션 모델 (rotation → translation → rotation)

상대 오도메트리를 추출할 때, 두 시점 사이의 움직임을 **세 단계**로 분해한다 (Probabilistic Robotics, Fig 5.7):

```
        δ_rot2
         ◜
    δ_trans
   ◜
 ●  δ_rot1
 (이전 pose)        →        (현재 pose)
```

| 단계 | 의미 |
|------|------|
| **δ_rot1** | 먼저 목표 방향으로 **회전** |
| **δ_trans** | 그 방향으로 **직선 이동** |
| **δ_rot2** | 도착 후 최종 자세로 **또 한 번 회전** |

수식 (motion_model_odometry):
```
δ_rot1  = atan2(ȳ'−ȳ, x̄'−x̄) − θ̄
δ_trans = √((x̄−x̄')² + (ȳ−ȳ')²)
δ_rot2  = θ̄' − θ̄ − δ_rot1
```
→ 이 세 값이 **상대 운동의 충분 통계량**. 알고리즘은 `p(xₜ | uₜ, xₜ₋₁)` (오도메트리로부터 현재 pose의 확률)을 계산한다.

> 핵심: 회전·직선·회전 세 파라미터는 **독립적인 노이즈**의 영향을 받는다고 가정 → 그래서 확률적으로 다룬다. ([[2026-06-16-amcl]]의 motion update가 이걸 씀)

## Part 2. 노이즈와 드리프트 — 오도메트리의 숙명

바퀴는 **미끄러지고(slip)**, 바닥은 **울퉁불퉁**하다. 그래서 엔코더가 "1m 갔다"고 해도 실제론 0.95m일 수 있다.

- 이 오차는 **계속 누적(accumulate)** 된다 → 시간이 갈수록 추정 위치가 실제와 벌어짐 = **드리프트(drift)**
- 그래서 오도메트리만으로는 **장시간/장거리 정확한 절대 위치를 알 수 없다.**
- 보정하려면 외부 기준(지도+LiDAR)이 필요 → [[2026-06-16-amcl]] (AMCL이 `map→odom`으로 이 드리프트를 잡아줌)

> 오도메트리 = **단기적으론 부드럽고 정확, 장기적으론 드리프트.** 이 성질이 아래 TF 구조의 이유가 된다.

---

## Part 3. ROS에서의 Odometry — odom frame

Odometry는 ROS에서 **`odom` frame**으로 구현되며, 엔코더 또는 IMU로 위치를 추정한다. (엔코더·IMU를 못 쓸 경우 **Visual Odometry** — 카메라 관측으로 odom 추정 가능)

### TF 프레임 구조
```
earth  →  map  →  odom  →  base_link
```
> `map`은 여러 개의 `odom`을 가질 수 있으나, `odom`은 단 하나의 `map`을 가진다.

| frame | 정의 | 특징 |
|-------|------|------|
| **earth** | 여러 로봇이 서로 다른 map 위에 존재하도록 묶는 최상위 | 멀티로봇/멀티빌딩용 |
| **map** | world fixed frame, earth에 고정된 전역 기준 | **불연속**(AMCL 보정 시 순간이동처럼 점프 가능) but 장기 global reference로 유용 |
| **odom** | world fixed frame, 연속적인 로컬 기준 | **연속적**(부드러움) but 드리프트로 장기 기준엔 부적합 |
| **base_link** | 로봇 자체의 위치를 나타내는 frame | odom/map을 통해 위치 추정 |

> 왜 `map`과 `odom`을 분리하나? → **map은 정확하지만 점프(불연속), odom은 부드럽지만 드리프트.** 둘을 분리해 "부드러움(odom)"과 "정확함(map)"을 모두 얻는다. 자세한 TF 메커니즘은 [[2026-06-16-tf]] 참고.

## Part 4. ROS2 개발자 보강 — 실무에서 알아야 할 것

- **표준 메시지**: `nav_msgs/Odometry` (pose + twist + covariance). 보통 `/odom` 토픽으로 발행.
- **누가 odom→base_link TF를 쏘나**: 바퀴 컨트롤러(`ros2_control` diff_drive_controller) 또는 가제보 플러그인, 혹은 `robot_localization` 노드.
- **센서 융합**: `robot_localization` 패키지의 **EKF/UKF** 노드가 휠 오도메트리 + IMU + (실외) GPS를 융합해 더 안정적인 odom을 만든다. → 칼만 필터의 실전 용례 ([[2026-06-16-filter]] 참고)
- **REP-105**: ROS 좌표계 표준(`map`/`odom`/`base_link` 의미와 관계)을 정의. 프레임 이름을 임의로 바꾸지 말 것.
- **odom은 navigation의 입력**: Nav2의 Controller·Velocity Smoother가 `odom`을 받아 제어 ([[2026-06-16-nav2-pkg]]).

## 한 줄 요약

> **Odometry는 엔코더·IMU로 "시작점 대비 얼마나 움직였나"를 계산하는 추측항법이다.** 회전–직선–회전 모델로 분해되고 노이즈가 누적돼 **드리프트**가 생기므로, 단기엔 정확하지만 장기엔 부정확하다. 그래서 ROS는 **연속적인 `odom`**과 **정확한 `map`** 프레임을 분리하고, AMCL/EKF가 그 사이 드리프트를 보정한다.
