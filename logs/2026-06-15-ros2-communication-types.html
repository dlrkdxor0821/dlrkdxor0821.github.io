ROS 통신 방식 3가지: Topic, Service, Action

ROS에서 노드끼리 데이터를 주고받는 방법은 3가지다. 셋을 고르는 기준은 딱 두 가지 질문으로 정리된다.

  • 응답을 받아야 하는가?
  • 작업이 오래 걸리는가?

이 두 질문의 답에 따라 어떤 방식을 쓸지 결정된다.

1. Topic — 단방향 스트리밍

데이터를 일방적으로 계속 흘려보내는 방식이다. 보내는 쪽(Publisher)은 받는 쪽이 있는지 신경 쓰지 않고, 받는 쪽(Subscriber)은 원하는 Topic을 구독하면 데이터를 받는다.

  • 방향: 단방향 (Publish → Subscribe)
  • 응답: 없음 (받았는지 확인 불가)
  • 연결: 다대다 (N:N)
  • 비유: 라디오 방송

적합한 경우

  • 라이다 / 카메라 / IMU 등 센서 데이터처럼 끊임없이 흐르는 값
  • /cmd_vel 같이 계속 갱신해서 보내는 명령

2. Service — 요청과 응답 (짧은 작업)

요청을 보내고 응답이 올 때까지 기다리는 방식이다. 응답을 받는 동안 코드가 멈춰서 대기(블로킹)한다.

  • 방향: 양방향 (Request → Response)
  • 응답: 있음 (대기)
  • 연결: 일대일
  • 비유: 전화 통화 (물어보고 답을 기다림)

적합한 경우

  • 파라미터 한 번 변경
  • 좌표 변환 한 번 계산
  • "현재 배터리 몇 %?" 같은 단발성 질의

주의: 응답이 올 때까지 멈추므로 오래 걸리는 작업에는 쓰면 안 된다. 작업이 끝날 때까지 코드가 묶이고, 중간 경과 확인도 취소도 불가능하다.

3. Action — 요청과 응답 (긴 작업)

Service와 비슷하게 요청-응답 구조지만, 오래 걸리는 작업을 위한 방식이다. 작업하는 동안 멈추지 않고(비동기), 중간 진행 상황을 받아보며, 도중에 취소할 수도 있다.

  • 방향: 양방향 + 중간 피드백
  • 구성: Goal(목표) → Feedback(진행 상황) → Result(결과), + Cancel(취소)
  • 연결: 일대일
  • 비유: 택배 주문 (주문 → 배송 추적 → 도착, 취소 가능)

적합한 경우

  • 내비게이션 (목표 지점까지 이동)
  • 로봇팔 궤적 동작
  • 오래 걸리는 배치 처리 작업

한눈에 비교

구분 Topic Service Action
방향 단방향 양방향 양방향 + 피드백
응답 없음 있음 (대기) 있음 (비동기)
작업 길이 계속 흐름 짧음 김
중간 경과 — 불가 가능
취소 — 불가 가능
연결 다대다 일대일 일대일
비유 라디오 방송 전화 통화 택배 주문

선택 기준 요약

응답이 필요 없고 데이터가 계속 흐른다       → Topic
응답이 필요하고 금방 끝난다                 → Service
응답이 필요하고 오래 걸린다 (경과·취소 필요) → Action

가장 흔한 실수: 오래 걸리는 작업을 Service로 만드는 것. 코드가 멈추고 진행 상황도 취소도 할 수 없으니, 이런 경우는 반드시 Action을 써야 한다.
