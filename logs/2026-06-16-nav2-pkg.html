Nav2 PKG 총정리 — 아키텍처·서버별 설정·Python 제어

Nav2#Nav2#Navigation2#BehaviorTree#costmap#RegulatedPurePursuit#lifecycle#BasicNavigator

Nav2 PKG 총정리 — 아키텍처·서버별 설정·Python 제어

ROS2 자율주행 내비게이션 스택 Nav2의 전체 구조와 서버별 핵심 설정, 그리고 Python으로 구동하는 법. (출처: PinkLab "ROS2 Nav2 PKG 설정"(48p) + "Python으로 Nav2 다뤄보기"(18p) + 개발자 보강) 흐름: 큰 그림 → Lifecycle/Launch/Composition → 6대 서버 → Costmap → Python 제어 → 실행 → 한 줄 요약

Part 0. 큰 그림 — Nav2 아키텍처

Nav2는 여러 독립 서버 노드를 BT Navigator가 지휘(orchestrate)하는 구조다.

                    ┌─────────────────────────────────────┐
   BT Navigator ──▶ │ Planner   Behavior   Waypoint        │
   (행동트리,최상위) │ Server    Server     Follower        │
                    │   │         (복구)      (경유지)       │
                    │ Smoother                              │
                    │ Server                                │
                    │   ▼                                   │
                    │ Controller Server ──cmd_vel_nav──▶    │
                    │   ▼                                   │
                    │ Velocity Smoother ──cmd_vel──▶ 로봇   │
                    └─────────────────────────────────────┘
        Lifecycle Manager가 모든 노드의 상태(Active/Inactive)를 일괄 관리
역할 노드 한 줄
최상위 지휘 BT Navigator Behavior Tree로 "무엇을 할지" 결정, 서버 호출
전역 경로 Planner Server 시작→목표 전역 경로 생성 (NavFn: Dijkstra/A*)
경로 후처리 Smoother Server 거친 경로를 부드럽게
경로 추종 Controller Server 경로 따라 cmd_vel 생성 (RPP)
복구 행동 Behavior Server Spin/BackUp/Wait 등 끼임 탈출
경유지 Waypoint Follower 여러 목표 순차 방문
속도 평활 Velocity Smoother 급가감속 제한 → 최종 cmd_vel

핵심: 각 서버는 ROS2 Action Server, BT Navigator는 Action Client로 이들을 부른다. Planner=ComputePathToPose, Controller=FollowPath, Behavior=Recovery.

Part 1. Lifecycle Node — Nav2의 뼈대

Nav2 핵심 노드는 모두 Managed(Lifecycle) Node. 상태를 가지고 순차적으로 켜진다.

Unconfigured ──configure()──▶ Inactive ──activate()──▶ Active ──shutdown()──▶ Finalized
상태 의미
Unconfigured 생성 직후, 파라미터 미로드
Inactive 구성 완료, 파라미터 보유, 발행/처리 미수행
Active 완전 가동 — 데이터 수신·처리·발행
Finalized 종료, 리소스 해제
  • Lifecycle Manager: 시스템의 지휘자. map_server, amcl, controller_server... 등을 순차적으로 configure→activate. autostart=True면 시작과 동시에 Active까지 자동 진입.
  • 실행 확인: 로그에 Managed nodes are active / Creating bond timer... 가 떠야 정상.
  • 왜? → 모든 노드가 준비된 뒤 동시에 동작하도록 보장 (경로 계획기만 떠 있고 맵이 안 떴는데 주행 시작하는 사고 방지).

Part 2. Launch 구조 & Composition

Launch 계층

bringup_launch.xml
 ├─ localization_launch.xml   → map_server + amcl  (lifecycle_nodes_local)
 └─ navigation_launch.xml     → planner/controller/behavior/bt/waypoint/smoother (lifecycle_nodes_nav)

주요 인자: namespace, map(yaml 경로), params_file(nav2_params.yaml), use_sim_time, autostart, use_composition, container_name.

Composition (Component Container)

여러 노드를 하나의 프로세스에 담아 실행 → 오버헤드 감소(프로세스 간 복사 대신 인트라프로세스 통신).

모드 설명
Composition (use_composition=True) load_composable_node로 컨테이너에 적재. CPU/메모리 절약
일반 노드 모드 (unless) 각 노드를 개별 프로세스로. 디버깅 편함
Isolated (component_container_isolated) 각 컴포넌트마다 전용 Executor(스레드풀) → 한 노드가 바빠도 다른 노드 안 막힘(blocking 방지)

컨테이너 내부 노드들은 /tf, /tf_static을 remap으로 공유 (네임스페이스 격리 시 충돌 방지). → [[2026-06-16-tf]]

Group 태그

<group>은 "실행"이 아니라 "설정 범위"의 단위. push-ros-namespace로 그룹 내 모든 토픽/서비스/액션 앞에 네임스페이스를 붙여 멀티로봇 격리. → [[2026-06-15-ros2-namespace]]

Part 3. Localization — map_server + AMCL

localization_launch.xml이 띄우는 두 노드:

  • map_server (nav2_map_server::MapServer): yaml_filename의 지도를 로드해 OccupancyGrid 토픽 발행.
  • amcl (nav2_amcl::AmclNode): LiDAR + 지도 매칭으로 위치 추정, map→odom TF 보정 발행.

AMCL의 동작 5단계(초기화→이동업데이트→센서업데이트→리샘플링→위치추정)와 파티클 필터 상세는 별도 정리: [[2026-06-16-amcl]]. 주요 파라미터 군: motion model(alpha1~5), frame/tf(base_frame_id/odom_frame_id/global_frame_id, tf_broadcast, transform_tolerance), 센서(laser_model_type: likelihood_field, max_beams), 입자수(min/max_particles), 업데이트 조건(update_min_d, update_min_a), set_initial_pose.

Part 4. Controller Server — 경로 추종 (RPP)

Controller server는 모터 제어기가 아니라 "로봇 제어기" — 전역 경로를 따라 로봇을 정밀하게 몰아 cmd_vel_nav를 만든다. 내부에 Local Costmap을 직접 host하여 실시간 장애물을 본다.

핵심 설정 (nav2_params.yaml)

controller_server:
  ros__parameters:
    controller_frequency: 20.0      # cmd_vel 생성 주기
    failure_tolerance: 0.3          # 경로 못 따라갈 때 허용 대기
    progress_checker:               # 진행 점검
      required_movement_radius: 0.5
      movement_time_allowance: 10.0 # 10초간 0.5m 못 가면 실패
    general_goal_checker:           # 도착 판정
      xy_goal_tolerance: 0.25       # 반경 0.25m
      yaw_goal_tolerance: 0.25      # 자세 0.25rad 안이면 성공
    FollowPath:
      plugin: "...RegulatedPurePursuitController"

Regulated Pure Pursuit (RPP)

경로 위 Lookahead Point(전방 주시점)를 따라가는 기하 기반 추종기 + 안전 규제.

단계 내용
1. Lookahead 선택 속도 비례 동적 거리 (use_velocity_scaled_lookahead_dist)
2. 곡률 계산 원호(Arc) 기반 조향각 산출
3. Regulation 급커브/장애물 근접 시 자동 감속 (안정성)
4. Rotate-to-Heading 경로 방향과 차이 크면 제자리 회전부터
5. 속도 명령 발행 가속도 제한 적용 → cmd_vel_nav

핵심 파라미터: lookahead_dist(0.6), min/max_lookahead_dist(0.3/0.9), use_collision_detection, use_rotate_to_heading, desired_linear_vel. 좁은 복도엔 rotate-to-heading이 큰 회전반경 충돌을 막아준다.

Lookahead 거리 = clamp(v×lookahead_time, min, max) — 빠를수록 멀리 본다(안정), 느릴수록 가까이(정밀). PID 추종과 비교하면 [[2026-06-16-pid-controller]]의 "추종 vs 회피" 계층과 같은 그림.

Part 5. Planner Server — 전역 경로 (NavFn)

planner_server:
  ros__parameters:
    expected_planner_frequency: 20.0
    planner_plugins: ["GridBased"]
    GridBased:
      plugin: "nav2_navfn_planner::NavfnPlanner"
      tolerance: 0.5          # 목표 도달 불가 시 근처 유효점
      use_astar: false        # false=Dijkstra, true=A*
      allow_unknown: true     # 미지 영역 통과 허용

Dijkstra vs A* (use_astar)

특성 Dijkstra (기본) A*
탐색 모든 방향 균등(BFS류) 목표 지향(휴리스틱)
효율 낮음(전체 스캔) 높음(필요 영역)
경로 품질 최단 보장(optimal) 휴리스틱 의존
적합 복잡한 미로, 소형 맵 개방 공간, 대형 맵

실내 작은 맵 = Dijkstra, 대형 물류창고 = A*(use_astar: true).

Planner는 Global Costmap을 내부 host (전체 정적 지도 기반, 낮은 빈도).

Part 6. Costmap & Layers — 비용 지도

로봇은 지도를 비용(cost) 으로 본다. 여러 레이어가 순서대로 쌓여 최종 costmap이 된다.

Map Server(OccupancyGrid) → Static Layer  (고정 벽/기둥)
LiDAR/Camera(scan)        → Obstacle Layer (실시간 동적 장애물, marking/clearing)
                          → Inflation Layer (장애물 주변 비용 팽창 = 가상 벽)
                          → 최종 costmap (경로 계획에 사용)
Layer 입력 역할
Static Map Server 변하지 않는 벽을 기본 골격으로
Obstacle/Voxel LiDAR/Depth(/scan) 실시간 장애물 추가/제거 (marking/clearing, raytrace_max_range)
Inflation 합성 costmap 장애물 주변에 마진(buffer) — 로봇이 벽에 너무 붙지 않게 (cost_scaling_factor, inflation_radius)
Global Costmap Local Costmap
host Planner Server Controller Server
기준 frame map odom
크기 전체 맵 로봇 주변 롤링 윈도우(3×3m, rolling_window)
빈도 낮음(1Hz) 높음(5Hz)
용도 전역 경로 실시간 회피
  • Footprint: 단순 반경(robot_radius) 대신 4꼭짓점 좌표(footprint)로 정확한 형상 정의 → 충돌 계산 정밀.
  • map_subscribe_transient_local: True: 지도 토픽 QoS(늦게 접속해도 latched 지도 수신).

Part 7. 나머지 서버들

Velocity Smoother

최종 cmd_vel의 급가감속을 제한(ramping)해 모터 충격·덜컹(jerk) 방지.

velocity_smoother:
  max_velocity: [0.25, 0.0, 1.5]   # [x, y, theta] (차동구동은 y=0)
  max_accel:    [2.5, 0.0, 3.2]
  max_decel:    [-2.5, 0.0, -3.2]
  deadband_velocity: [0.0, 0.0, 0.0]  # 미세 속도 무시 → 노이즈 제거
  velocity_timeout: 1.0               # 입력 끊기면 정지(안전)

입력 cmd_vel_nav → 출력 cmd_vel로 remap. (deadband/slew 개념은 [[2026-06-16-pid-controller]]와 동일 도구함)

Smoother Server

Planner가 만든 거친 경로를 부드럽게 후처리 (Planner 실행 후, Controller 추종 전).

Behavior Server — 복구 행동

BT가 끼임/실패를 감지하면 호출하는 복구 플러그인 5종: Spin(제자리 회전 재스캔), BackUp(후진), DriveOnHeading(경로 없이 직진), Wait(대기), AssistedTeleop(충돌방지 수동조종). 기준 프레임: local_frame: odom, global_frame: map.

Waypoint Follower

여러 경유지를 순차 방문. stop_on_failure(한 점 실패 시 다음 진행 여부), WaitAtWaypoint(도착마다 waypoint_pause_duration만큼 정지 — 사진 촬영·재고 스캔 등).

BT Navigator — 최상위 오케스트레이터

Behavior Tree(BT)를 로드·실행해 전체 미션의 논리 흐름을 관장. 직접 로봇을 제어하지 않고 "무엇을 할지" 결정해 Planner/Controller/Behavior를 Action Client로 호출·모니터링.

bt_navigator:
  global_frame: map
  robot_base_frame: base_link
  odom_topic: odom
  bt_loop_duration: 10   # BT tick 주기(ms)

Part 8. Python으로 Nav2 제어하기

목표 전송 방식은 두 가지 — Action(BasicNavigator) vs Topic(goal_pose).

① Action 방식 — BasicNavigator (권장)

양방향 통신: 피드백·결과 수신, 취소 가능.

import rclpy
from geometry_msgs.msg import PoseStamped
from nav2_simple_commander.robot_navigator import BasicNavigator, TaskResult

rclpy.init()
nav = BasicNavigator()
nav.waitUntilNav2Active()          # lifecycle Active 대기

goal = PoseStamped()
goal.header.frame_id = 'map'       # 목표는 map 기준
goal.header.stamp = nav.get_clock().now().to_msg()
goal.pose.position.x, goal.pose.position.y = 0.0, 1.0
goal.pose.orientation.z, goal.pose.orientation.w = q[2], q[3]  # yaw→quat

nav.goToPose(goal)                 # 내부적으로 NavigateToPose 액션 전송 (비동기)

while not nav.isTaskComplete():     # 진행 모니터링
    fb = nav.getFeedback()
    print(f"남은 거리: {fb.distance_remaining:.2f} m")
    # 타임아웃 취소
    # if elapsed > CANCEL_AFTER_SEC: nav.cancelTask(); break

print(nav.getResult())             # SUCCEEDED / CANCELED / FAILED
  • goToPose()는 단순 함수 같지만 내부적으로 Action Client 생성 → Nav2 스택 전체 구동의 트리거.
  • 내부 흐름: BT → Planner(전역경로) → Controller(로컬추종) → Recovery(복구).
  • 사용 가능 액션: ros2 action list → /navigate_to_pose, /navigate_through_poses, /follow_waypoints, /spin, /backup...

② Topic 방식 — /goal_pose (간단, Fire-and-Forget)

피드백/결과 없이 목표만 던지고 끝. RViz "2D Goal Pose"와 같은 방식.

self.pub = self.create_publisher(PoseStamped, "/goal_pose", 10)
# 발행 전 구독자 확인 (DDS late-join 대비)
while self.pub.get_subscription_count() == 0 and wait < timeout:
    rclpy.spin_once(self, timeout_sec=0.1); wait += 0.1
self.pub.publish(msg)              # 여러 번 발행 권장(놓침 방지)

⚠️ get_subscription_count()로 Nav2가 듣고 있는지 확인 후 발행 — DDS는 연결에 시간이 걸려 너무 빨리 쏘면 유실. ([[2026-06-15-ros2-dds]])

Action vs Topic

Action(BasicNavigator) Topic(goal_pose)
통신 양방향(피드백·결과·취소) 단방향(Fire&Forget)
신뢰성 높음(상태 추적) 낮음(던지고 끝)
코드 다소 복잡 매우 간단
용도 실무 미션 제어 빠른 테스트

(통신 방식 비교는 [[2026-06-15-ros2-communication-types]])

# 터미널1: 시뮬레이터 (창고 월드 + 로봇)

# 터미널2: Nav2 Bringup (맵 + lifecycle 활성화)

# 터미널3: RViz 시각화
ros2 launch pinky_navigation nav2_view.launch.xml
#   → Navigation/Localization 패널 Active(초록), Map·TF 정상 표시 확인

한 줄 요약

Nav2는 "BT Navigator가 지휘하고, Lifecycle Manager가 켜고, 각 서버(Planner·Controller·Behavior·Smoother·Waypoint)가 한 역할씩 맡는" 자율주행 스택이다. Planner가 전역경로(Dijkstra/A*)를, Controller(RPP)가 Local Costmap을 보며 실시간 추종을, Velocity Smoother가 부드러운 cmd_vel을 만든다. 위치추정은 AMCL([[2026-06-16-amcl]]), 좌표는 TF([[2026-06-16-tf]])가 받친다. Python에선 BasicNavigator.goToPose()(양방향) 또는 /goal_pose 토픽(간단)으로 목표를 준다.
