2:T2fe7,<h1>Nav2 Planner·Controller 플러그인 심층 분석</h1>
<blockquote>
<p>Planner Server와 Controller Server는 <strong>플러그인 방식</strong>이라 알고리즘을 갈아끼울 수 있다.
각 서버별로 <strong>요약표 → 플러그인별 (작동원리 · 언제 · 장단점)</strong> 순서로 분석한다.
(전체 Nav2 구조는 [[2026-06-16-nav2-pkg]], 추종 제어 감각은 [[2026-06-16-pid-controller]])</p>
</blockquote>

<h3>▸ NavFn (<code>nav2_navfn_planner::NavfnPlanner</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 목표 지점에서 <strong>비용 파동(wavefront)을 전 격자로 전파</strong>(Dijkstra식)한 뒤, 그 비용 경사(gradient)를 거꾸로 타고 내려와 경로를 뽑는다. <code>use_astar:true</code>면 목표 지향 휴리스틱(A*)으로 탐색 범위를 줄인다. → [[2026-06-16-nav2-pkg]] Part 5.</li>
<li><strong>언제</strong>: 제자리 회전 되는 <strong>원형/차동 로봇 + 실내 소형 맵</strong>. 빠르게 굴리고 싶을 때 첫 선택.</li>
<li><strong>장점</strong>: 단순·빠름·안정적, 튜닝할 게 거의 없음, 미지영역 통과(<code>allow_unknown</code>).</li>
<li><strong>단점</strong>: <strong>로봇 기구학(회전반경) 완전 무시</strong> → 차량형엔 못 씀. 경로가 격자에 묶여 살짝 각지고 벽에 붙는 경향.</li>
</ul>
<h3>▸ Smac 2D (<code>nav2_smac_planner::SmacPlanner2D</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 정통 <strong>2D A</strong>* (8-connected) + 비용함수(거리+장애물 페널티). NavFn보다 현대적 구현이고 <strong>경로 스무더 내장</strong>.</li>
<li><strong>언제</strong>: 원형/홀로노믹 로봇에서 NavFn보다 매끈한 경로를 원할 때.</li>
<li><strong>장점</strong>: 빠르면서 경로 품질이 NavFn보다 좋음.</li>
<li><strong>단점</strong>: 여전히 <strong>점(point) 로봇 가정</strong> — 방향·기구학 미반영.</li>
</ul>
<h3>▸ Smac Hybrid-A* (<code>nav2_smac_planner::SmacPlannerHybrid</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 상태에 <strong>방향 θ를 추가</strong>(x, y, θ). 차량 운동모델(<strong>최소 회전반경</strong>, 전진/후진)을 지키는 <strong>motion primitive</strong>로만 노드를 확장하고, 목표 근처에선 <strong>Dubins/Reeds-Shepp 곡선</strong>으로 해석적 연결(analytic expansion).</li>
<li><strong>언제</strong>: <strong>Ackermann/car-like(앞바퀴 조향)</strong>, 비홀로노믹, <strong>후진이 필요한</strong> 로봇.</li>
<li><strong>장점</strong>: 실제로 <strong>주행 가능한(kinematically feasible)</strong> 경로 — 그대로 따라가면 된다.</li>
<li><strong>단점</strong>: 무겁고 느림, 파라미터 많고 튜닝 까다로움(회전반경·primitive 해상도 등).</li>
</ul>
<h3>▸ Smac State Lattice (<code>nav2_smac_planner::SmacPlannerLattice</code>)</h3>
<ul>
<li><strong>작동원리</strong>: **오프라인에서 미리 생성한 motion primitive 집합(lattice)**으로 그래프 탐색. 로봇의 **실제 footprint(형상)**까지 충돌 검사에 반영.</li>
<li><strong>언제</strong>: <strong>비원형·특이 형상</strong> 로봇, 특정 기동 제약(예: 특수 차량)을 정밀히 지켜야 할 때.</li>
<li><strong>장점</strong>: 임의 형상·기구학을 가장 정밀하게 반영.</li>
<li><strong>단점</strong>: primitive 파일 생성·설정이 복잡, 무거움. 일반 로봇엔 과함(overkill).</li>
</ul>
<h3>▸ Theta* (<code>nav2_theta_star_planner::ThetaStarPlanner</code>)</h3>
<ul>
<li><strong>작동원리</strong>: A* 변형. 부모를 격자 이웃으로 제한하지 않고 <strong>시야(line-of-sight)가 트이면 직접 연결</strong> → 격자 계단식 지그재그 없는 <strong>임의 각도 직선</strong> 경로.</li>
<li><strong>언제</strong>: <strong>개방 공간</strong>에서 자연스러운 직선 경로를 원할 때.</li>
<li><strong>장점</strong>: 짧고 자연스러운 경로(불필요한 꺾임 제거).</li>
<li><strong>단점</strong>: LOS 체크 비용, 기구학 무시.</li>
</ul>
<blockquote>
<p><strong>Planner 선택 의사결정</strong>: 제자리 회전 됨(차동·원형) → <strong>NavFn/Smac2D</strong> · 차처럼 조향 → <strong>Smac Hybrid</strong> · 특이 형상 → <strong>Lattice</strong> · 넓은 개방공간 직선 → <strong>Theta*</strong>.</p>
</blockquote>

<h3>▸ DWB (<code>dwb_core::DWBLocalPlanner</code>)</h3>
<ul>
<li><strong>작동원리</strong>: <strong>Dynamic Window</strong> — 현재 속도에서 도달 가능한 <code>(v, ω)</code> 속도 후보들을 샘플링 → 각각 짧은 미래 궤적을 시뮬레이션 → <strong>여러 critic</strong>(목표거리·경로정렬·장애물·진동 등)으로 점수화 → 최고점 속도를 명령.</li>
<li><strong>언제</strong>: 복잡한 환경에서 <strong>행동을 세밀하게 조형</strong>하고 싶을 때(차동·홀로노믹).</li>
<li><strong>장점</strong>: critic 가중치로 유연하게 튜닝, 오래 검증된 안정성.</li>
<li><strong>단점</strong>: <strong>critic 가중치가 많아 튜닝이 어려움</strong>, 후진/제자리 회전 약하고 좁은 곳에서 보수적.</li>
</ul>
<h3>▸ RPP (<code>nav2_regulated_pure_pursuit_controller::RegulatedPurePursuitController</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 경로 위 <strong>Lookahead Point</strong>를 향하는 <strong>원호 곡률</strong>을 계산해 따라감 + 안전 규제(급커브·장애물 근접 시 감속, 큰 방향차면 rotate-to-heading). → [[2026-06-16-nav2-pkg]] Part 4 + [[2026-06-16-pid-controller]].</li>
<li><strong>언제</strong>: <strong>차동구동</strong>에서 단순하고 견고한 추종을 원할 때, 좁은 복도.</li>
<li><strong>장점</strong>: 단순·직관·견고, <strong>튜닝 쉬움</strong>, 예측 가능한 거동.</li>
<li><strong>단점</strong>: <strong>능동적 동적 장애물 회피가 약함</strong>(경로 추종 위주) — 회피는 costmap+감속에 의존.</li>
</ul>
<h3>▸ MPPI (<code>nav2_mppi_controller::MPPIController</code>)</h3>
<ul>
<li><strong>작동원리</strong>: <strong>Model Predictive Path Integral</strong> — 매 주기 <strong>수천 개의 제어 시퀀스를 노이즈로 샘플링</strong> → 각각 미래를 롤아웃 → 비용함수(critic: 경로추종·장애물·속도 등)로 <strong>가중 평균</strong> → 최적 제어 도출. CPU 병렬로 동작.</li>
<li><strong>언제</strong>: <strong>부드러움·고품질 주행</strong>, 동적 장애물이 많은 환경, 다양한 기구학(차동·옴니·Ackermann).</li>
<li><strong>장점</strong>: 매끄럽고 <strong>예측적인 똑똑한 회피</strong>, 기구학 유연, 요즘 추천 1순위.</li>
<li><strong>단점</strong>: <strong>계산 부하 높음</strong>(CPU 여유 필요), 파라미터 많아 튜닝에 학습곡선, 제어주파수 확보 필수.</li>
</ul>
<h3>▸ Rotation Shim (<code>nav2_rotation_shim_controller::RotationShimController</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 새 경로 시작 시 로봇 방향이 경로와 크게 어긋나면 <strong>먼저 제자리 회전</strong>시키고, 정렬되면 <strong>본 컨트롤러(DWB/MPPI 등)로 위임</strong>하는 <strong>래퍼</strong>.</li>
<li><strong>언제</strong>: 출발 시 <strong>큰 호를 그리며 휘어 나가는 것</strong>을 막고 싶을 때(특히 DWB와 조합).</li>
<li><strong>장점</strong>: 자연스러운 출발, 초기 충돌 위험↓.</li>
<li><strong>단점</strong>: <strong>단독 사용 불가</strong> — 반드시 primary controller와 조합 전제.</li>
</ul>
<h3>▸ Graceful (<code>nav2_graceful_controller::GracefulController</code>)</h3>
<ul>
<li><strong>작동원리</strong>: 운동학 기반 **부드러운 제어 법칙(control law)**으로 목표 <strong>자세(pose)에 우아하게 수렴</strong>.</li>
<li><strong>언제</strong>: <strong>도킹·정밀 자세 정렬</strong>(목표 방향까지 딱 맞춰야 할 때).</li>
<li><strong>장점</strong>: 매우 부드럽고 우아한 접근.</li>
<li><strong>단점</strong>: 일반 장거리 주행/적극 회피엔 부적합.</li>
</ul>
<blockquote>
<p>(참고: <strong>TEB</strong>(Timed Elastic Band)는 core Nav2가 아닌 커뮤니티 플러그인이지만 차량형에서 많이 쓴다.)</p>
</blockquote>
<blockquote>
<p><strong>Controller 선택 의사결정</strong>: 단순·견고(차동) → <strong>RPP</strong> · 고품질·동적환경 → <strong>MPPI</strong> · 행동 세밀튜닝 → <strong>DWB</strong>(+<strong>Rotation Shim</strong>) · 도킹 → <strong>Graceful</strong>.</p>
</blockquote>

<h2>Part 3. 교체 방법 (params의 <code>plugin:</code> 한 줄)</h2>
<pre><code class="language-yaml">planner_server:
  ros__parameters:
    planner_plugins: [&quot;GridBased&quot;]
    GridBased:
      plugin: &quot;nav2_smac_planner::SmacPlannerHybrid&quot;   # ← 이 줄만 바꾸면 교체

controller_server:
  ros__parameters:
    controller_plugins: [&quot;FollowPath&quot;]
    FollowPath:
      plugin: &quot;nav2_mppi_controller::MPPIController&quot;     # ← 여기도
</code></pre>

<h2>한 줄 요약</h2>
<blockquote>
<p><strong>Planner는 &quot;기구학&quot;으로 고른다</strong>(점 로봇이면 NavFn/Smac2D/Theta*, 방향·회전반경 있으면 Smac Hybrid/Lattice). <strong>Controller는 &quot;주행 품질 vs 계산 비용&quot;으로 고른다</strong>(가벼움·단순=RPP, 똑똑함·부드러움=MPPI, 세밀튜닝=DWB). Planner가 만든 길을 Controller가 따라가니 <strong>둘은 짝을 맞춰</strong> 고른다(예: Smac Hybrid + MPPI = 차량형 고품질 조합).</p>
</blockquote>
