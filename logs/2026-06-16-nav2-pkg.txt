<h2>Part 0. 큰 그림 — Nav2 아키텍처</h2>
<p>Nav2는 여러 <strong>독립 서버 노드</strong>를 <strong>BT Navigator</strong>가 지휘(orchestrate)하는 구조다.</p>
<pre><code>                    ┌─────────────────────────────────────┐
   BT Navigator ──▶ │ Planner   Behavior   Waypoint        │
   (행동트리,최상위) │ Server    Server     Follower        │
                    │   │         (복구)      (경유지)       │
                    │ Smoother                              │
                    │ Server                                │
                    │   ▼                                   │
                    │ Controller Server ──cmd_vel_nav──▶    │
                    │   ▼                                   │
                    │ Velocity Smoother ──cmd_vel──▶ 로봇   │
                    └─────────────────────────────────────┘
        Lifecycle Manager가 모든 노드의 상태(Active/Inactive)를 일괄 관리
</code></pre>

<th>노드</th>
<th>한 줄</th>
</tr>
</thead>

<td>BT Navigator</td>
<td>Behavior Tree로 &quot;무엇을 할지&quot; 결정, 서버 호출</td>
</tr>
<tr>
<td><strong>전역 경로</strong></td>
<td>Planner Server</td>

<tr>

<h2>Part 1. Lifecycle Node — Nav2의 뼈대</h2>
<p>Nav2 핵심 노드는 모두 <strong>Managed(Lifecycle) Node</strong>. 상태를 가지고 순차적으로 켜진다.</p>
<pre><code>Unconfigured ──configure()──▶ Inactive ──activate()──▶ Active ──shutdown()──▶ Finalized
</code></pre>
<table>
<thead>
<tr>
<th>상태</th>
<th>의미</th>
</tr>
</thead>
<tbody><tr>
<td>Unconfigured</td>
<td>생성 직후, 파라미터 미로드</td>
</tr>
<tr>
<td>Inactive</td>
<td>구성 완료, 파라미터 보유, 발행/처리 미수행</td>
</tr>
<tr>
<td><strong>Active</strong></td>
<td>완전 가동 — 데이터 수신·처리·발행</td>
</tr>
<tr>
<td>Finalized</td>
<td>종료, 리소스 해제</td>
</tr>
</tbody></table>
<ul>
<li><strong>Lifecycle Manager</strong>: 시스템의 지휘자. <code>map_server</code>, <code>amcl</code>, <code>controller_server</code>... 등을 <strong>순차적으로 configure→activate</strong>. <code>autostart=True</code>면 시작과 동시에 Active까지 자동 진입.</li>
<li>실행 확인: 로그에 <code>Managed nodes are active</code> / <code>Creating bond timer...</code> 가 떠야 정상.</li>
<li>왜? → 모든 노드가 <strong>준비된 뒤 동시에</strong> 동작하도록 보장 (경로 계획기만 떠 있고 맵이 안 떴는데 주행 시작하는 사고 방지).</li>
</ul>

<h2>Part 2. Launch 구조 &amp; Composition</h2>
<h3>Launch 계층</h3>
<pre><code>bringup_launch.xml
 ├─ localization_launch.xml   → map_server + amcl  (lifecycle_nodes_local)
 └─ navigation_launch.xml     → planner/controller/behavior/bt/waypoint/smoother (lifecycle_nodes_nav)
</code></pre>
<p>주요 인자: <code>namespace</code>, <code>map</code>(yaml 경로), <code>params_file</code>(nav2_params.yaml), <code>use_sim_time</code>, <code>autostart</code>, <code>use_composition</code>, <code>container_name</code>.</p>
<h3>Composition (Component Container)</h3>
<p>여러 노드를 <strong>하나의 프로세스</strong>에 담아 실행 → <strong>오버헤드 감소</strong>(프로세스 간 복사 대신 인트라프로세스 통신).</p>
<table>
<thead>
<tr>
<th>모드</th>
<th>설명</th>
</tr>
</thead>
<tbody><tr>
<td><strong>Composition</strong> (<code>use_composition=True</code>)</td>
<td><code>load_composable_node</code>로 컨테이너에 적재. CPU/메모리 절약</td>
</tr>
<tr>
<td>일반 노드 모드 (<code>unless</code>)</td>
<td>각 노드를 개별 프로세스로. 디버깅 편함</td>
</tr>
<tr>
<td><strong>Isolated</strong> (<code>component_container_isolated</code>)</td>
<td>각 컴포넌트마다 <strong>전용 Executor</strong>(스레드풀) → 한 노드가 바빠도 다른 노드 안 막힘(blocking 방지)</td>
</tr>
</tbody></table>
<blockquote>
<p>컨테이너 내부 노드들은 <code>/tf</code>, <code>/tf_static</code>을 <strong>remap</strong>으로 공유 (네임스페이스 격리 시 충돌 방지). → [[2026-06-16-tf]]</p>
</blockquote>
<h3>Group 태그</h3>
<p><code>&lt;group&gt;</code>은 &quot;실행&quot;이 아니라 <strong>&quot;설정 범위&quot;의 단위</strong>. <code>push-ros-namespace</code>로 그룹 내 모든 토픽/서비스/액션 앞에 네임스페이스를 붙여 <strong>멀티로봇 격리</strong>. → [[2026-06-15-ros2-namespace]]</p>
<hr>
<h2>Part 3. Localization — map_server + AMCL</h2>
<p><code>localization_launch.xml</code>이 띄우는 두 노드:</p>
<ul>
<li><strong>map_server</strong> (<code>nav2_map_server::MapServer</code>): <code>yaml_filename</code>의 지도를 로드해 <code>OccupancyGrid</code> 토픽 발행.</li>
<li><strong>amcl</strong> (<code>nav2_amcl::AmclNode</code>): LiDAR + 지도 매칭으로 위치 추정, <code>map→odom</code> TF 보정 발행.</li>
</ul>
<p>AMCL의 동작 5단계(초기화→이동업데이트→센서업데이트→리샘플링→위치추정)와 파티클 필터 상세는 별도 정리: <strong>[[2026-06-16-amcl]]</strong>.
주요 파라미터 군: motion model(<code>alpha1~5</code>), frame/tf(<code>base_frame_id</code>/<code>odom_frame_id</code>/<code>global_frame_id</code>, <code>tf_broadcast</code>, <code>transform_tolerance</code>), 센서(<code>laser_model_type: likelihood_field</code>, <code>max_beams</code>), 입자수(<code>min/max_particles</code>), 업데이트 조건(<code>update_min_d</code>, <code>update_min_a</code>), <code>set_initial_pose</code>.</p>
<hr>
<h2>Part 4. Controller Server — 경로 추종 (RPP)</h2>
<blockquote>
<p><strong>Controller server는 모터 제어기가 아니라 &quot;로봇 제어기&quot;</strong> — 전역 경로를 따라 로봇을 정밀하게 몰아 <code>cmd_vel_nav</code>를 만든다. 내부에 <strong>Local Costmap</strong>을 직접 host하여 실시간 장애물을 본다.</p>
</blockquote>
<h3>핵심 설정 (nav2_params.yaml)</h3>
<pre><code class="language-yaml">controller_server:
  ros__parameters:
    controller_frequency: 20.0      # cmd_vel 생성 주기
    failure_tolerance: 0.3          # 경로 못 따라갈 때 허용 대기
    progress_checker:               # 진행 점검
      required_movement_radius: 0.5
      movement_time_allowance: 10.0 # 10초간 0.5m 못 가면 실패
    general_goal_checker:           # 도착 판정
      xy_goal_tolerance: 0.25       # 반경 0.25m
      yaw_goal_tolerance: 0.25      # 자세 0.25rad 안이면 성공
    FollowPath:
      plugin: &quot;...RegulatedPurePursuitController&quot;
</code></pre>
<h3>Regulated Pure Pursuit (RPP)</h3>
<p>경로 위 <strong>Lookahead Point</strong>(전방 주시점)를 따라가는 기하 기반 추종기 + 안전 규제.</p>
<table>
<thead>
<tr>
<th>단계</th>
<th>내용</th>
</tr>
</thead>
<tbody><tr>
<td>1. Lookahead 선택</td>
<td>속도 비례 동적 거리 (<code>use_velocity_scaled_lookahead_dist</code>)</td>
</tr>
<tr>
<td>2. 곡률 계산</td>
<td>원호(Arc) 기반 조향각 산출</td>
</tr>
<tr>
<td>3. <strong>Regulation</strong></td>
<td>급커브/장애물 근접 시 <strong>자동 감속</strong> (안정성)</td>
</tr>
<tr>
<td>4. Rotate-to-Heading</td>
<td>경로 방향과 차이 크면 <strong>제자리 회전</strong>부터</td>
</tr>
<tr>
<td>5. 속도 명령 발행</td>
<td>가속도 제한 적용 → <code>cmd_vel_nav</code></td>
</tr>
</tbody></table>
<p>핵심 파라미터: <code>lookahead_dist</code>(0.6), <code>min/max_lookahead_dist</code>(0.3/0.9), <code>use_collision_detection</code>, <code>use_rotate_to_heading</code>, <code>desired_linear_vel</code>. 좁은 복도엔 rotate-to-heading이 큰 회전반경 충돌을 막아준다.</p>
<blockquote>
<p>Lookahead 거리 = <code>clamp(v×lookahead_time, min, max)</code> — 빠를수록 멀리 본다(안정), 느릴수록 가까이(정밀). PID 추종과 비교하면 [[2026-06-16-pid-controller]]의 &quot;추종 vs 회피&quot; 계층과 같은 그림.</p>
</blockquote>

<h2>Part 5. Planner Server — 전역 경로 (NavFn)</h2>
<pre><code class="language-yaml">planner_server:
  ros__parameters:
    expected_planner_frequency: 20.0
    planner_plugins: [&quot;GridBased&quot;]
    GridBased:
      plugin: &quot;nav2_navfn_planner::NavfnPlanner&quot;
      tolerance: 0.5          # 목표 도달 불가 시 근처 유효점
      use_astar: false        # false=Dijkstra, true=A*
      allow_unknown: true     # 미지 영역 통과 허용
</code></pre>

<h2>Part 7. 나머지 서버들</h2>
<h3>Velocity Smoother</h3>
<p>최종 <code>cmd_vel</code>의 <strong>급가감속을 제한</strong>(ramping)해 모터 충격·덜컹(jerk) 방지.</p>
<pre><code class="language-yaml">velocity_smoother:
  max_velocity: [0.25, 0.0, 1.5]   # [x, y, theta] (차동구동은 y=0)
  max_accel:    [2.5, 0.0, 3.2]
  max_decel:    [-2.5, 0.0, -3.2]
  deadband_velocity: [0.0, 0.0, 0.0]  # 미세 속도 무시 → 노이즈 제거
  velocity_timeout: 1.0               # 입력 끊기면 정지(안전)
</code></pre>
<p>입력 <code>cmd_vel_nav</code> → 출력 <code>cmd_vel</code>로 remap. (deadband/slew 개념은 [[2026-06-16-pid-controller]]와 동일 도구함)</p>
<h3>Smoother Server</h3>
<p>Planner가 만든 거친 경로를 <strong>부드럽게 후처리</strong> (Planner 실행 후, Controller 추종 전).</p>
<h3>Behavior Server — 복구 행동</h3>
<p>BT가 끼임/실패를 감지하면 호출하는 <strong>복구 플러그인 5종</strong>:
<code>Spin</code>(제자리 회전 재스캔), <code>BackUp</code>(후진), <code>DriveOnHeading</code>(경로 없이 직진), <code>Wait</code>(대기), <code>AssistedTeleop</code>(충돌방지 수동조종). 기준 프레임: <code>local_frame: odom</code>, <code>global_frame: map</code>.</p>
<h3>Waypoint Follower</h3>
<p>여러 경유지를 순차 방문. <code>stop_on_failure</code>(한 점 실패 시 다음 진행 여부), <code>WaitAtWaypoint</code>(도착마다 <code>waypoint_pause_duration</code>만큼 정지 — 사진 촬영·재고 스캔 등).</p>
<h3>BT Navigator — 최상위 오케스트레이터</h3>
<p>Behavior Tree(BT)를 로드·실행해 전체 미션의 논리 흐름을 관장. 직접 로봇을 제어하지 않고 <strong>&quot;무엇을 할지&quot;</strong> 결정해 Planner/Controller/Behavior를 Action Client로 호출·모니터링.</p>
<pre><code class="language-yaml">bt_navigator:
  global_frame: map
  robot_base_frame: base_link
  odom_topic: odom
  bt_loop_duration: 10   # BT tick 주기(ms)
</code></pre>

<h2>Part 8. Python으로 Nav2 제어하기</h2>
<p>목표 전송 방식은 두 가지 — <strong>Action(BasicNavigator)</strong> vs <strong>Topic(goal_pose)</strong>.</p>
<h3>① Action 방식 — BasicNavigator (권장)</h3>
<p>양방향 통신: 피드백·결과 수신, 취소 가능.</p>
<pre><code class="language-python">import rclpy
from geometry_msgs.msg import PoseStamped
from nav2_simple_commander.robot_navigator import BasicNavigator, TaskResult

rclpy.init()
nav = BasicNavigator()
nav.waitUntilNav2Active()          # lifecycle Active 대기

goal = PoseStamped()
goal.header.frame_id = &#39;map&#39;       # 목표는 map 기준
goal.header.stamp = nav.get_clock().now().to_msg()
goal.pose.position.x, goal.pose.position.y = 0.0, 1.0
goal.pose.orientation.z, goal.pose.orientation.w = q[2], q[3]  # yaw→quat

nav.goToPose(goal)                 # 내부적으로 NavigateToPose 액션 전송 (비동기)

while not nav.isTaskComplete():     # 진행 모니터링
    fb = nav.getFeedback()
    print(f&quot;남은 거리: {fb.distance_remaining:.2f} m&quot;)
    # 타임아웃 취소
    # if elapsed &gt; CANCEL_AFTER_SEC: nav.cancelTask(); break

print(nav.getResult())             # SUCCEEDED / CANCELED / FAILED
</code></pre>
<ul>
<li><code>goToPose()</code>는 단순 함수 같지만 내부적으로 <strong>Action Client 생성 → Nav2 스택 전체 구동</strong>의 트리거.</li>
<li>내부 흐름: BT → Planner(전역경로) → Controller(로컬추종) → Recovery(복구).</li>
<li>사용 가능 액션: <code>ros2 action list</code> → <code>/navigate_to_pose</code>, <code>/navigate_through_poses</code>, <code>/follow_waypoints</code>, <code>/spin</code>, <code>/backup</code>...</li>
</ul>
<h3>② Topic 방식 — /goal_pose (간단, Fire-and-Forget)</h3>
<p>피드백/결과 없이 목표만 던지고 끝. RViz &quot;2D Goal Pose&quot;와 같은 방식.</p>
<pre><code class="language-python">self.pub = self.create_publisher(PoseStamped, &quot;/goal_pose&quot;, 10)
# 발행 전 구독자 확인 (DDS late-join 대비)
while self.pub.get_subscription_count() == 0 and wait &lt; timeout:
    rclpy.spin_once(self, timeout_sec=0.1); wait += 0.1
self.pub.publish(msg)              # 여러 번 발행 권장(놓침 방지)
</code></pre>
<blockquote>
<p>⚠️ <code>get_subscription_count()</code>로 Nav2가 듣고 있는지 확인 후 발행 — DDS는 연결에 시간이 걸려 너무 빨리 쏘면 유실. ([[2026-06-15-ros2-dds]])</p>
</blockquote>

<h2>한 줄 요약</h2>
<blockquote>
<p><strong>Nav2는 &quot;BT Navigator가 지휘하고, Lifecycle Manager가 켜고, 각 서버(Planner·Controller·Behavior·Smoother·Waypoint)가 한 역할씩 맡는&quot; 자율주행 스택이다.</strong> Planner가 전역경로(Dijkstra/A*)를, Controller(RPP)가 Local Costmap을 보며 실시간 추종을, Velocity Smoother가 부드러운 <code>cmd_vel</code>을 만든다. 위치추정은 AMCL([[2026-06-16-amcl]]), 좌표는 TF([[2026-06-16-tf]])가 받친다. Python에선 <strong>BasicNavigator.goToPose()</strong>(양방향) 또는 <strong>/goal_pose 토픽</strong>(간단)으로 목표를 준다.</p>
</blockquote>
