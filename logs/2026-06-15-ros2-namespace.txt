2:T1cbb,<h1>ROS2 네임스페이스 — 다중로봇 정리</h1>
<h2>1. 정의</h2>
<p><strong>네임스페이스(namespace)</strong> = 노드·토픽·서비스·액션 이름 앞에 붙는 <strong>경로 접두사</strong>. 파일 시스템의 폴더처럼 이름을 묶어 <strong>충돌을 방지</strong>한다.</p>
<pre><code>robot1 네임스페이스 →  노드 /robot1/controller
                      토픽 cmd_vel → /robot1/cmd_vel
</code></pre>
<p>이름 해석 규칙:</p>
<table>
<thead>
<tr>
<th>종류</th>
<th>표기</th>
<th>결과 (NS=robot1)</th>
</tr>
</thead>
<tbody><tr>
<td>상대(relative)</td>
<td><code>cmd_vel</code></td>
<td><code>/robot1/cmd_vel</code> ✅ 네임스페이스 적용</td>
</tr>
<tr>
<td>절대(absolute)</td>
<td><code>/tf</code></td>
<td><code>/tf</code> ❌ 그대로 (NS 무시)</td>
</tr>
<tr>
<td>private</td>
<td><code>~/param</code></td>
<td><code>/robot1/노드명/param</code></td>
</tr>
</tbody></table>
<blockquote>
<p>한 줄: <strong>상대 이름만 자동으로 갈린다. 절대 이름(<code>/tf</code> 등)은 안 갈린다.</strong></p>
</blockquote>
<hr>
<h2>2. 구현 방안 — 공통 vs 분리</h2>
<h3>🔵 공통 (전 로봇이 1개 공유 / root namespace)</h3>
<table>
<thead>
<tr>
<th>자원</th>
<th>이유</th>
</tr>
</thead>
<tbody><tr>
<td><strong>시뮬레이터 서버</strong> (Gazebo)</td>
<td>물리 월드는 하나</td>
</tr>
<tr>
<td><strong>맵 서버 (map_server)</strong></td>
<td>보통 모두 같은 지도 공유</td>
</tr>
<tr>
<td><strong><code>map</code> 프레임</strong></td>
<td>모든 로봇의 공통 최상위 좌표계 (← <code>frame_prefix</code> 안 붙이는 유일한 프레임)</td>
</tr>
<tr>
<td><strong>RViz / RMF 관제</strong></td>
<td>전체를 한눈에 보는 도구</td>
</tr>
</tbody></table>
<h3>🟢 분리 (로봇마다 별도 네임스페이스)</h3>
<table>
<thead>
<tr>
<th>자원</th>
<th>분리 방법</th>
</tr>
</thead>
<tbody><tr>
<td>노드·상대 토픽</td>
<td><code>PushRosNamespace</code> (자동)</td>
</tr>
<tr>
<td><code>/tf</code>·<code>/tf_static</code></td>
<td>상대경로 remap (수동)</td>
</tr>
<tr>
<td>프레임 ID (<code>base_link</code>,<code>odom</code>)</td>
<td><code>frame_prefix</code> (수동)</td>
</tr>
<tr>
<td>파라미터 속 프레임</td>
<td><code>RewrittenYaml</code> 치환 (수동)</td>
</tr>
<tr>
<td>spawn 엔티티 이름·초기 위치</td>
<td>로봇마다 유니크 인자</td>
</tr>
</tbody></table>
<h3>한눈 요약 — &quot;4종 세트&quot;</h3>
<blockquote>
<p><strong>분리 = <code>PushRosNamespace</code> + <code>/tf</code> remap + <code>frame_prefix</code> + <code>RewrittenYaml</code></strong>
(namespace 한 줄만으론 절대 안 됨)</p>
</blockquote>
<h3>TF 트리 결과</h3>
<pre><code>map ─┬─ robot1/odom ─ robot1/base_link ─ robot1/lidar
     └─ robot2/odom ─ robot2/base_link ─ robot2/lidar
</code></pre>

<h2>3. 구체적 코드 구현</h2>
<h3>(a) 핵심 빌딩블록 — 로봇 1대를 &quot;봉투&quot;로 감싸기</h3>
<pre><code class="language-python">from launch.actions import GroupAction
from launch_ros.actions import Node, PushRosNamespace
from launch.actions import IncludeLaunchDescription
from nav2_common.launch import RewrittenYaml

def robot_group(name, x, y, nav2_launch, params_file):
    ns = name
    tf_remap = [(&#39;/tf&#39;, &#39;tf&#39;), (&#39;/tf_static&#39;, &#39;tf_static&#39;)]   # ② TF 분리

    # ③ 파라미터 속 프레임 치환
    params = RewrittenYaml(
        source_file=params_file,
        root_key=ns,
        param_rewrites={
            &#39;robot_base_frame&#39;: f&#39;{ns}/base_link&#39;,
            &#39;odom_frame&#39;:       f&#39;{ns}/odom&#39;,
            &#39;global_frame&#39;:     &#39;map&#39;,           # map은 공유 → 접두사 X
        },
        convert_types=True,
    )

    return GroupAction([
        PushRosNamespace(ns),                    # ① 그룹 전체 /ns/ 상속

        # robot_state_publisher — 프레임 접두사 + tf remap
        Node(
            package=&#39;robot_state_publisher&#39;,
            executable=&#39;robot_state_publisher&#39;,
            parameters=[{&#39;frame_prefix&#39;: f&#39;{ns}/&#39;}],   # ③ 프레임 ID 분리
            remappings=tf_remap,
        ),

        # 시뮬에 스폰 — 유니크 이름 + 초기 위치
        Node(
            package=&#39;ros_gz_sim&#39;, executable=&#39;create&#39;,
            arguments=[&#39;-name&#39;, ns, &#39;-x&#39;, str(x), &#39;-y&#39;, str(y), &#39;-topic&#39;, &#39;robot_description&#39;],
        ),

        # nav2 풀스택 include — namespace + 치환된 params 전달
        IncludeLaunchDescription(
            nav2_launch,
            launch_arguments={
                &#39;namespace&#39;:    ns,
                &#39;use_namespace&#39;:&#39;true&#39;,
                &#39;params_file&#39;:  params,
                &#39;use_sim_time&#39;: &#39;true&#39;,
            }.items(),
        ),
    ])
</code></pre>
<h3>(b) bringup.py — 공통 1번 + 로봇 N개 루프</h3>
<pre><code class="language-python">from launch import LaunchDescription

def generate_launch_description():
    robots = [
        {&#39;name&#39;: &#39;robot1&#39;, &#39;x&#39;: 0.0, &#39;y&#39;: 0.0},
        {&#39;name&#39;: &#39;robot2&#39;, &#39;x&#39;: 2.0, &#39;y&#39;: 0.0},
    ]

    ld = []
    # 🔵 공통 자원 (root namespace, 1번만)
    ld.append(gazebo_server_launch())     # 시뮬레이터
    ld.append(map_server_node())          # 공유 맵

    # 🟢 로봇별 그룹 (루프로 N개)
    for r in robots:
        ld.append(robot_group(
            r[&#39;name&#39;], r[&#39;x&#39;], r[&#39;y&#39;],
            nav2_launch=nav2_bringup_launch,
            params_file=&#39;nav2_params.yaml&#39;,
        ))

    return LaunchDescription(ld)
</code></pre>

ros2 topic list     # /robot1/cmd_vel, /robot2/cmd_vel
ros2 run tf2_tools view_frames   # map 아래 robot1/*, robot2/* 두 갈래
</code></pre>

<h2>4. 흔한 함정 체크리스트</h2>
<ul>
<li><input disabled="" type="checkbox"> <code>/tf</code> remap 빠뜨림 → 두 로봇 TF가 <code>/tf</code>에서 충돌 (가장 흔한 실수)</li>
<li><input disabled="" type="checkbox"> <code>frame_prefix</code>만 하고 params 프레임 치환 안 함 → nav2가 <code>base_link</code> 못 찾음</li>
<li><input disabled="" type="checkbox"> <code>map</code> 프레임에 접두사 붙임 → 로봇 간 좌표 공유 깨짐</li>
<li><input disabled="" type="checkbox"> spawn 엔티티 이름 중복 → 두 번째 로봇 스폰 실패</li>
</ul>

<h2>5. 대안: 네임스페이스 대신 DDS 도메인 분리</h2>
<p>완전히 독립된 로봇(서로 통신 불필요)이면 네임스페이스 대신 <code>ROS_DOMAIN_ID</code>를 다르게 줘서 네트워크 레벨에서 통째로 격리할 수도 있다.</p>
<pre><code class="language-bash">ROS_DOMAIN_ID=1 ros2 launch robot bringup.py   # robot1
ROS_DOMAIN_ID=2 ros2 launch robot bringup.py   # robot2
</code></pre>
<p>단 이러면 한 관제(RMF 등)에서 여러 로봇을 동시에 못 본다 → 협동·관제가 필요하면 네임스페이스 방식이 맞다.</p>
